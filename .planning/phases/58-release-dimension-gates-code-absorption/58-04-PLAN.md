**If quality-post-release.yml's workflow_run trigger is the issue:**

The `workflow_run` event is famously brittle:
- Only triggers on workflows in the default branch's HEAD (per GH docs).
- GITHUB_TOKEN-pushed tags do NOT chain workflows (P56 SURPRISES.md row 2).
- The trigger may delay 1-5 minutes; testing via workflow_dispatch is the proxy.

If workflow_dispatch on quality-post-release.yml works but a real release.yml run does NOT chain to it: pivot per the rule below.

**Pivot pattern (single workflow with cadence-matrix):**

Rewrite quality-post-release.yml:

```yaml
name: quality-post-release

on:
  schedule:
    # Backup cadence: if workflow_run is brittle, weekly cron catches drift.
    - cron: '0 10 * * 1'   # Monday 10:00 UTC (1h after quality-weekly)
  workflow_dispatch:
  workflow_run:
    workflows: ["release"]
    types: [completed]

permissions:
  contents: read

concurrency:
  group: quality-post-release-${{ github.event.workflow_run.id || github.event_name }}
  cancel-in-progress: false

jobs:
  post-release-gates:
    if: ${{ github.event_name != 'workflow_run' || github.event.workflow_run.conclusion == 'success' }}
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      # ... (same as before — checkout, python, rust, cargo-binstall, run runner)
```

The schedule trigger is a backup that fires weekly; if workflow_run successfully chains, the post-release runner fires within 1-5 min of release.yml; if not, the next Monday's cron catches the drift. Either way, the catalog gets graded eventually.

## Trust Boundaries

| Boundary | Description |
|---|---|
| committed workflow YAML ↔ GH Actions runtime | The committed YAML defines what GH Actions runs; tampering visible in code review |
| GH Actions secrets ↔ workflow steps | Both new workflows use minimal secrets (none beyond GITHUB_TOKEN); no third-party secrets |
| README badge URL ↔ external rendering | GitHub auto-generates the badge SVG; URL is project-controlled |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-58-04-01 | Tampering | workflow YAML edited to skip the runner | mitigate | banned-words-lint runs on commit; future P63 cohesion adds a structure-dim row asserting the workflow exists |
