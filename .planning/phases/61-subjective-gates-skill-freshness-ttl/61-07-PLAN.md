Wave G — **POLISH-SUBJECTIVE broaden-and-deepen sweep**. Per the prompt's broaden-and-deepen directive ("the user wants the codebase pristine across all dimensions; don't ship the bare minimum") + the REQUIREMENTS.md POLISH-SUBJECTIVE row ("dispatch the unbiased subagent for [3 rubrics] AT LEAST ONCE; fix any P0/P1 findings in the same phase; remaining P2 → fixed / waived with expiry / v0.12.1 carry-forward"):

Three activities, in order:

1. **Dispatch all 3 rubrics** (Task 1). Run `bash .claude/skills/reposix-quality-review/dispatch.sh --all-stale --force` from the SKILL.md Path A flow (Task tool from Claude session). 3 subagents dispatched in parallel per OP-2. Each persists a real artifact JSON.

2. **Triage findings + fix in-phase OR file as v0.12.1 carry-forward** (Task 2). Read each artifact's `rationale` + `evidence_files`. For each P0/P1 finding (score < 7): edit README.md / docs/index.md surgically; re-dispatch the rubric; confirm PASS. For P2 findings or unfixable-in-phase: file v0.12.1 carry-forward in MIGRATE-03 + extend the catalog row's waiver to until=2026-07-26.

3. **Confirm catalog flip + runner GREEN** (Task 3). Run `python3 quality/runners/run.py --cadence pre-release` and confirm the 2 pre-release rows are PASS (not WAIVED). Run `--cadence weekly` and confirm the 1 weekly row is PASS or PARTIAL or carries a v0.12.1 waiver. Verifier-subagent dispatch in Wave H grades the GREEN verdict.

**Why is Wave G `autonomous: false`?**

- **Subagent dispatch is the human-or-checkpoint moment.** The dispatcher invokes Task / claude -p with the rubric prompts + source files. The output is unpredictable (subagents may produce different verdicts on the same inputs). The Wave G executor needs to read the verdicts + decide whether to fix-in-phase or carry-forward — this is a judgment call.
- **Source-file edits.** README.md + docs/index.md are user-facing prose. Edits to fix a cold-reader rubric finding (or install-positioning, or headline-numbers) require careful prose work; not autonomous task work.
- **The checkpoint moment** is the rubric-verdict triage: the executor reviews the 3 artifacts, decides per-row what to do, then proceeds.

**Why dispatch in parallel per OP-2?**

The 3 rubrics are independent: they share no state, they read mostly-disjoint file sets (cold-reader: README + docs/index; install-positioning: README + docs/index install sections; headline-numbers: README + docs/index + benchmarks/), and the Anthropic API supports concurrent subagent dispatch. Dispatching sequentially adds 3-15 minutes; dispatching in parallel adds 1-5 minutes. OP-2 is the right call.

The `cargo memory budget` rule does NOT apply because subjective rubrics are pure-prose Claude reads — no cargo, no rustc, no rust-analyzer. 3 parallel subagents are safe.

**Why broaden-and-deepen, not just dispatch-and-pass?**

The prompt's broaden-and-deepen directive is explicit: "don't ship the bare minimum — POLISH-SUBJECTIVE means actually run the rubrics and fix what they catch, not just scaffold and waive."

The Wave G executor's job is NOT just to dispatch the rubrics + persist the artifacts. It is to:
- Run the rubrics with real inputs.
- Read every finding the subagent flags.
- Decide which findings are P0/P1 (fix in-phase) vs P2 (fix-or-waive).
- Actually fix the P0/P1 findings (surgical edits to README.md / docs/index.md).
- Document the P2 findings in v0.12.1 carry-forward.
- Re-dispatch after fixes to confirm the catalog rows flip to PASS.

This is the close-the-loop principle (OP-1) for prose: render → reload-as-stranger → ask "does this land?" → fix → re-render → re-ask.

**Pivot rules for the dispatch:**

- **Subagent returns wildly inconsistent verdicts on the same input** (per Q2 in `.planning/research/v0.12.0/open-questions-and-deferrals.md`): re-dispatch the rubric twice; if delta > 2pp on the score, flag as PARTIAL. Document in SURPRISES.md. The rubric's `expected.asserts` already accommodates this (the catalog locks "score >= 7 PASS"; not "exactly 8.5 PASS").

- **A rubric flags a P0 finding that requires significant work** (e.g. headline-numbers finds 3 uncited numbers; the agent needs to write 3 benchmarks/ artifacts to cite): file as v0.12.1 carry-forward + extend waiver. Do NOT panic-rewrite.

- **All 3 rubrics PASS on first dispatch**: that's the happy path. Catalog rows flip to PASS at the next runner sweep. Wave G's only commit is the artifact-persistence commit.

**Commit strategy:**

- Commit 1: `chore(p61): POLISH-SUBJECTIVE -- dispatch all 3 rubrics + persist artifacts`. Body cites: 3 artifacts written; per-rubric score + verdict; runner re-grade flipped catalog rows to PASS (cite specific row IDs + scores).
- Commit 2 (only if P0/P1 findings fixed): `fix(p61): cold-reader/install-positioning/headline-numbers findings -- POLISH-SUBJECTIVE Wave G`. Body cites the specific findings + the source-file edits.
- Commit 3 (only if waivers extended): `docs(p61): extend subjective waivers for v0.12.1 MIGRATE-03 carry-forwards`. Body cites the specific waivers + the v0.12.1 entries.

Output: 1-3 commits depending on findings.
