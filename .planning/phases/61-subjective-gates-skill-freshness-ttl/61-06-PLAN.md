**Why does the pre-release workflow trigger AT THE SAME TIME as release.yml (parallel) instead of release.yml waiting for quality-pre-release?**

GitHub Actions cross-workflow `needs:` declarations are NOT supported. The only ways to chain across workflows are:

- **`workflow_run` trigger**: workflow B fires only when workflow A completes. Latency 1-2 minutes; A's failures show up as "B was not triggered" which is confusing UX.
- **Required Checks** (branch protection): the maintainer enforces "release.yml only runs if quality-pre-release passes" at the GitHub branch-protection level, not in YAML. Requires admin perms + can be bypassed.
- **`gh api` polling** in release.yml's first job: release.yml polls quality-pre-release's status until completion, then proceeds OR fails. Adds 2-5 min to release.yml; brittle to API rate limits.
- **Single composite workflow**: merge quality-pre-release.yml INTO release.yml as a job-level `needs:` declaration. This is the cleanest GH-Actions-idiomatic chaining, but tightly couples the catalog verdict to the release pipeline.

For v0.12.0, the simplest path: **parallel execution + alerting**. quality-pre-release.yml triggers on the same tag push; if it fails, the maintainer manually invalidates the release (deletes the GH release object + the artifacts). The catalog verdict becomes a soft-gate, not a hard-gate. The hard-gate variant (composite workflow OR workflow_run chaining) is a v0.12.1 carry-forward via MIGRATE-03.

**Why does release.yml get the dependency declaration even though chaining is deferred?**

Adding a comment + a placeholder dependency to release.yml signals the contract to future maintainers: "this is meant to wait for quality-pre-release, but v0.12.0 ships parallel due to GH Actions limitations." The placeholder is structurally important even if functionally a no-op. The pivot rule below covers the implementation choice.

**Why does the runner need a docstring update?**

Wave B added `is_stale`. The runner's `is_in_scope` + `run_row` integration was generic — STALE handling fires for any cadence with the right row state. But the pre-release cadence is the ONLY cadence where STALE detection materially affects gates (subjective rows are pre-release or weekly; weekly is non-blocking by P2 status). The docstring update on `quality/runners/run.py` explicitly notes: "pre-release cadence is where the freshness-TTL gate matters; STALE subjective rows block release without a new artifact." This makes the contract discoverable to a future agent reading the runner.

**Why no auto-dispatch on STALE in v0.12.0?**

Auto-dispatch from CI would require:
- The GH Actions runner to have Anthropic API auth (not currently the case).
- The runner to know how to invoke `claude /reposix-quality-review` from a non-Claude environment.
- Trust boundary: the runner becomes a long-lived agent invocation surface, which raises threat-model concerns (per CLAUDE.md threat model + the autonomous-execution-protocol).

The v0.12.0 impl: pre-release workflow FAILS on STALE rows with a clear error message pointing the maintainer at `bash .claude/skills/reposix-quality-review/dispatch.sh --all-stale --force` (which the maintainer runs LOCALLY in a Claude session). Wave G demonstrates this end-to-end. The auto-dispatch is filed as v0.12.1 carry-forward in MIGRATE-03.

**.github/workflows/quality-pre-release.yml shape (model on quality-weekly.yml + quality-post-release.yml; <=80 LOC):**

```yaml
name: Quality (pre-release)

on:
  push:
    tags:
      - 'v*'
      - 'reposix-cli-v*'
  workflow_dispatch:
    inputs:
      tag:
        description: 'Tag to pre-release-verify (defaults to current ref)'
        required: false

permissions:
  contents: read
  checks: write   # for catalog-verdict GH Check Run output

jobs:
  pre-release:
    name: Pre-release catalog verification
    runs-on: ubuntu-24.04
    timeout-minutes: 15
    env:
      GH_TOKEN: ${{ github.token }}
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: Run quality runner --cadence pre-release
        id: runner
        run: |
          set -euo pipefail
          python3 quality/runners/run.py --cadence pre-release | tee /tmp/pre-release-runner.log
          echo "exit_code=$?" >> $GITHUB_OUTPUT

      - name: Generate verdict
        if: always()
        run: |
          python3 quality/runners/verdict.py --cadence pre-release || true
          if [ -f quality/reports/verdicts/pre-release/*.md ]; then
            cat quality/reports/verdicts/pre-release/*.md | tail -100 >> $GITHUB_STEP_SUMMARY
          fi

      - name: Fail on STALE subjective rows
        if: failure() && contains(steps.runner.outputs.exit_code, '1')
        run: |
          echo "::error::Quality pre-release verdict RED. STALE or FAIL rows present."
          echo "::error::Run locally: bash .claude/skills/reposix-quality-review/dispatch.sh --all-stale --force"
          echo "::error::Then re-run this workflow via 'gh workflow run quality-pre-release.yml --ref <tag>'"
          exit 1
```
