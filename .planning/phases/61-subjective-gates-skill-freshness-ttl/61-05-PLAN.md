<objective>
Wave E — implement the 2 remaining rubrics: **install-positioning** + **headline-numbers-sanity**. Unlike the cold-reader rubric (Wave D used doc-clarity-review subprocess), these 2 use the inline-dispatch pattern: a custom subagent prompt + source-files list, dispatched via Path A (Task tool from Claude session) or Path B (claude -p subprocess fallback). Output: canonical artifact JSON with rubric-specific evidence fields.

Three deliverables:

1. **`rubrics/install-positioning.md`** — verbatim prompt body. <=80 LOC. Asks the subagent to read README.md install section + docs/index.md install section, grade whether the hero install snippet leads with a package manager (cargo binstall / brew install / cargo install / pip install / npm install / go install) BEFORE any git-clone+build snippet. Numeric score 1-10. Returns: score / verdict / rationale / `install_paths_observed` list.

2. **`rubrics/headline-numbers-sanity.md`** — verbatim prompt body. <=120 LOC. Asks the subagent to read README.md + docs/index.md + benchmarks/ + docs/benchmarks/ files, list every headline number, cross-check against the source-of-truth artifacts, grade drift. Numeric score 1-10. Returns: score / verdict / rationale / `headline_numbers_table` (number | file:line | source-file | drift_pp).

3. **`lib/dispatch_inline_subagent.sh`** — rubric dispatcher. Routes per RUBRIC_ID argument; constructs subagent prompt + sources list per rubric; invokes Path A (Task tool from a Claude session that has Task) OR Path B (claude -p subprocess); parses the subagent JSON output; calls lib/persist_artifact.py to write the canonical artifact. <=120 LOC.

**Why split install-positioning + headline-numbers from cold-reader?**

- Cold-reader uses an EXISTING skill (doc-clarity-review). Subprocess invocation = `claude /doc-clarity-review --prompt ... README.md docs/index.md`. The output format is fixed by doc-clarity-review.
- Install-positioning + headline-numbers are CUSTOM rubrics. No existing skill implements them. The dispatcher constructs a custom subagent prompt + sources list.
- Same pattern; 2 different impls. Wave D handled the doc-clarity-review one; Wave E handles the inline-custom ones.

**Why include `benchmarks/` AND `docs/benchmarks/` in headline-numbers-sanity?**

Per `quality/catalogs/subjective-rubrics.json` Wave A row 3, headline-numbers-sanity sources cite both `benchmarks/` (raw bench scripts + outputs) and `docs/benchmarks/` (rendered bench docs). Both are source-of-truth for different number classes:
- `benchmarks/fixtures/` = raw bench inputs (token counts, timing measurements).
- `docs/benchmarks/` = rendered bench results (the percentages and absolute numbers users see).

If the README cites "89.1% token economy", the source-of-truth lives in `docs/benchmarks/` (or whatever the bench page reports). If the bench landing page itself drifted from the raw data, that's a different rubric run.

**Why the rubric-specific evidence fields?**

The canonical artifact JSON shape is fixed (`{ts, rubric_id, score, verdict, rationale, evidence_files, dispatched_via, asserts_passed, asserts_failed, stale}`). But each rubric has additional structured evidence the subagent can produce:

- install-positioning: `install_paths_observed` = `["README.md hero leads with: cargo binstall reposix-cli", "docs/index.md hero leads with: cargo binstall reposix-cli"]`. Useful for Wave G's POLISH-SUBJECTIVE: if the rubric flags FAIL, the agent reads this list to know exactly which files have which leading commands.

- headline-numbers-sanity: `headline_numbers_table` = `[{number: "89.1%", source_in_doc: "README.md:23", source_of_truth: "docs/benchmarks/token-economy.md:45", drift_pp: 0.0}]`. Each row tells you: the number, where it appears in the hero, where the source-of-truth lives, the percentage-point drift.

These fields are added to the artifact JSON via lib/persist_artifact.py's CLI extension (Wave D added argparse; Wave E adds 2 optional `--evidence-extra <key>=<json>` args, OR the rubric-specific evidence is written as part of the rationale paragraph and parsed by the runner re-grade — DECISION: add a `--evidence-extra` JSON-string arg to persist_artifact.py CLI; the dispatcher passes the rubric-specific JSON; persist_artifact.py merges it into the artifact).

**Path A vs Path B dispatch in dispatch_inline_subagent.sh:**

- **Path A**: the dispatcher detects it's running in a Claude session with Task tool available (env var or stdin probe). Constructs the subagent prompt (rubric body + source files inlined or referenced), dispatches via `Task({subagent_type: 'general-purpose', prompt: ...})`. The subagent returns JSON; the dispatcher parses + persists.

- **Path B**: the dispatcher is invoked from a runner subprocess (no Task tool). Falls back to `claude -p '<prompt with files attached>'` (the same pattern doc-clarity-review uses). Captures stdout, parses for the JSON output (or the structured-prose verdict, depending on what claude -p returns), persists.

For Wave E, the implementation defaults to Path B (claude -p) because the runner subprocess path is the always-available case. Path A is preferred when the dispatcher is invoked from a SKILL.md flow that has Task — Wave G's POLISH-SUBJECTIVE invocation goes through this path.

**rubrics/install-positioning.md (verbatim prompt body, <=80 LOC):**

```markdown
You are reviewing the INSTALL section of one or more documentation files in complete isolation. You have NOT seen the codebase or any other files. Read only what is provided.

Your job: grade whether the install copy gives a published-tool first impression OR a build-from-source first impression.

A "published-tool first impression" is when the FIRST install command in the hero / install section is a package-manager command:
- `cargo install <name>` or `cargo binstall <name>` (Rust packages)
- `brew install <name>` (Homebrew)
- `pip install <name>` (Python packages)
- `npm install <name>` or `npm i <name>` (Node packages)
- `go install <path>` (Go packages)
- `apt install <name>`, `dnf install <name>`, `pacman -S <name>` (system packages)
- `curl <installer-url> | sh` (one-liner installer that resolves to a published binary)
- `winget install <name>`, `choco install <name>` (Windows package managers)

A "build-from-source first impression" is when the FIRST install command is:
- `git clone <repo>` followed by a build step
- `cargo build` or `make` or `npm build` against a checkout
- ANY command that requires the user to have the source tree before they can install

Build-from-source paths are FINE as a fallback option. They are NOT fine as the lead. A user landing on the install section should see "you can install this with one command" before they see "or you can clone and build."

Report the following sections:

## Install paths observed

For EACH file you read, list the FIRST install command in the install section, in the order they appear. Cite line numbers.

## Lead command type

For each file: is the first install command a package-manager command, a one-liner installer, or a build-from-source step?

## Friction points

Specific issues with the install copy: missing target list (which platforms? which architectures?), unclear version pinning, broken-looking command syntax, etc.

## Verdict

Numeric score 1-10:
- 10: every file's hero leads with a package-manager command, lists supported targets, names a binstall/binary fallback explicitly.
- 7-9: leads with package-manager command but missing one of: target list, fallback path, or version pinning guidance.
- 4-6: leads with package-manager command but the build-from-source path is given equal billing OR appears confusingly close to the lead.
- 1-3: leads with `git clone && build` (worse first impression for a published tool).

Rate: <integer 1-10>
Verdict: <CLEAR if score >= 7 / NEEDS-WORK if 4-6 / CONFUSING if 1-3>
Rationale: <one paragraph>
```

**rubrics/headline-numbers-sanity.md (verbatim prompt body, <=120 LOC):**

```markdown
You are reviewing documentation files in complete isolation. You have NOT seen the codebase or any other files. Read only what is provided.

Your job: cross-check every "headline number" in the user-facing hero / overview / landing pages against the benchmark or measurement files in the same input set.

A "headline number" is any quantitative claim that appears in the hero / overview / landing copy, of these shapes:
- Percentages: "89.1% reduction", "40% faster", "99.9% uptime"
- Multipliers: "10x faster", "3x smaller"
- Absolute numbers: "1ms latency", "60 seconds", "200 lines"
- Ratios: "1:1000", "1 in 100"
- Time-bounded numbers: "in 5 minutes", "under 1 second"

For each headline number you find:

1. Where in the hero / overview does it appear? (file path + line number)
2. Is there a citation in the same paragraph (link, footnote, "(see benchmarks/foo.md)")? If yes: read the cited source.
3. Is there a benchmark / measurement file in the input set whose name suggests it covers this number? (e.g. "token-economy.md" likely covers a "token economy" claim)
4. If the source-of-truth file exists: does the number in the source-of-truth match the headline?
   - Percentages: drift <= 2 percentage points = OK; drift > 2pp = flag.
   - Multipliers: drift <= 10% = OK; drift > 10% = flag.
   - Absolute numbers: within stated confidence interval = OK; outside CI = flag.
5. If no source-of-truth file is found: flag as "uncited headline number".

Report the following sections:

## Headline numbers found

A table for every number found:

| Number | File:line in hero | Source-of-truth file | Drift | Status |
|---|---|---|---|---|
| 89.1% | README.md:23 | docs/benchmarks/token-economy.md:45 | 0.0pp | OK |
| 10x faster | docs/index.md:12 | (none found) | - | UNCITED |

## Friction points

What's confusing about how the numbers are presented? Are they cited inline? Are the source-of-truth artifacts easy to find from the citation?

## Verdict

Numeric score 1-10:
- 10: every headline number cites a source-of-truth artifact AND drift is within tolerance.
- 7-9: every headline number cites a source-of-truth AND drift is within tolerance, but 1-2 numbers have stale citations (3-5pp drift) — fix-soon territory.
- 4-6: 1-2 headline numbers are uncited OR drift exceeds tolerance for 1+ numbers.
- 1-3: 3+ uncited numbers OR major drift (>5pp on a percentage, >25% on a multiplier).

Rate: <integer 1-10>
Verdict: <CLEAR if score >= 7 / NEEDS-WORK if 4-6 / CONFUSING if 1-3>
Rationale: <one paragraph>
```

**lib/dispatch_inline_subagent.sh sketch (target <=120 LOC):**

```bash
#!/usr/bin/env bash
# .claude/skills/reposix-quality-review/lib/dispatch_inline_subagent.sh
# Rubric dispatcher for inline-dispatch rubrics (install-positioning + headline-numbers-sanity).
# Path A (Task tool) preferred; Path B (claude -p) fallback.
set -euo pipefail

RUBRIC_ID="${1:?--rubric required}"
REPO_ROOT="$(cd "$(dirname "$0")/../../../.." && pwd)"
SKILL_DIR="$REPO_ROOT/.claude/skills/reposix-quality-review"

# Map rubric id -> prompt path + source-file glob
case "$RUBRIC_ID" in
    "subjective/install-positioning")
        PROMPT="$SKILL_DIR/rubrics/install-positioning.md"
        SOURCES=("$REPO_ROOT/README.md" "$REPO_ROOT/docs/index.md")
        ;;
    "subjective/headline-numbers-sanity")
        PROMPT="$SKILL_DIR/rubrics/headline-numbers-sanity.md"
        SOURCES=("$REPO_ROOT/README.md" "$REPO_ROOT/docs/index.md")
        # Also include benchmarks/ + docs/benchmarks/ as source-of-truth
        for f in "$REPO_ROOT/docs/benchmarks/"*.md "$REPO_ROOT/benchmarks/"*.md "$REPO_ROOT/benchmarks/fixtures/"*; do
            [ -f "$f" ] && SOURCES+=("$f")
        done
        ;;
    *)
        echo "ERROR: unknown rubric: $RUBRIC_ID" >&2
        exit 1
        ;;
esac

# Path B (claude -p subprocess) — the always-available case
TMPLOG="$(mktemp /tmp/inline-dispatch-XXXXXX.log)"
trap 'rm -f "$TMPLOG"' EXIT

# Build the subagent prompt: rubric body + the file paths
SUBAGENT_PROMPT="$(cat "$PROMPT")"$'\n\nFiles to review:\n'
for f in "${SOURCES[@]}"; do
    SUBAGENT_PROMPT+="$f"$'\n'
done

# Invoke claude -p (or fail-safe stub for non-claude environments)
if command -v claude >/dev/null 2>&1; then
    claude -p "$SUBAGENT_PROMPT" "${SOURCES[@]}" 2>&1 | tee "$TMPLOG" || {
        # Fallback artifact on subprocess failure
        python3 "$SKILL_DIR/lib/persist_artifact.py" \
            --rubric-id "$RUBRIC_ID" \
            --score 0 \
            --verdict CONFUSING \
            --rationale "claude -p subprocess failed; see $TMPLOG" \
            --evidence-files "${SOURCES[@]##*/}" \
            --dispatched-via "claude-p-failed" \
            --asserts-failed "claude -p subprocess exited non-zero"
        exit 1
    }
else
    # No Claude CLI in env (CI without Anthropic auth) — emit fallback
    python3 "$SKILL_DIR/lib/persist_artifact.py" \
        --rubric-id "$RUBRIC_ID" \
        --score 0 \
        --verdict CONFUSING \
        --rationale "claude CLI not in PATH; cannot dispatch subagent" \
        --evidence-files "${SOURCES[@]##*/}" \
        --dispatched-via "claude-cli-not-in-path" \
        --asserts-failed "claude CLI not available"
    exit 1
fi

# Parse the verdict (the subagent's structured prose ends with "Rate: <int>")
SCORE="$(grep -E '^Rate: [0-9]+' "$TMPLOG" | tail -1 | sed 's/^Rate: //;s/ *$//')"
VERDICT="$(grep -E '^Verdict: ' "$TMPLOG" | tail -1 | sed 's/^Verdict: //;s/ *$//')"
[ -z "$SCORE" ] && SCORE=0
[ -z "$VERDICT" ] && VERDICT="CONFUSING"

# Extract rationale
RATIONALE="$(awk '/^Rationale:/,/^$/' "$TMPLOG" | tail -n +2 | head -10 | tr '\n' ' ')"
[ -z "$RATIONALE" ] && RATIONALE="(no rationale extracted)"

# Persist canonical artifact
python3 "$SKILL_DIR/lib/persist_artifact.py" \
    --rubric-id "$RUBRIC_ID" \
    --score "$SCORE" \
    --verdict "$VERDICT" \
    --rationale "$RATIONALE" \
    --evidence-files "${SOURCES[@]##*/}" \
    --dispatched-via "claude-p-inline"

# Exit per threshold
if [ "$SCORE" -ge 7 ]; then exit 0
elif [ "$SCORE" -ge 4 ]; then exit 2
else exit 1
fi
```
