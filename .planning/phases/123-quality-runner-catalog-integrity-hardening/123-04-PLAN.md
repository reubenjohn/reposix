  <behavior>
    - Test 1: committed (git HEAD) catalog has row X with `status: PASS`; a fresh `--persist` run
      grades X as `FAIL`; WITHOUT `--allow-downgrade`, `save_catalog` is NOT called for that
      catalog file, and `main()` prints a message containing the row id, `"PASS"`, `"FAIL"`, and
      the literal substring `"--allow-downgrade"`.
    - Test 2: same setup, but the invocation passes `--allow-downgrade` -> `save_catalog` IS called
      and the FAIL is persisted (the flag restores the old behavior).
    - Test 3: committed status `WAIVED` -> fresh grade `FAIL` -> also blocked without
      `--allow-downgrade` (WAIVED counts as green, not just PASS).
    - Test 4: a row with the SAME id NOT present in the git-HEAD committed catalog (brand-new row,
      e.g. straight out of 123-01's catalog-first commit) is never blocked, regardless of its
      fresh grade.
    - Test 5: committed `PASS`, fresh grade STILL `PASS` (no change) -> not blocked (only an actual
      downgrade trips it).
    - Test 6 (SC2 REGRESSION-VS-TTL SEMANTICS â€” deadlock prevention): committed `PASS`, fresh grade
      `NOT-VERIFIED` (simulating a freshness-TTL expiry, a missing-verifier demotion, an env-skip,
      or an exit-75 NOT-VERIFIED convention exit â€” ANY cause) -> NOT blocked, WITHOUT
      `--allow-downgrade`, and `save_catalog` IS called / no refusal message is printed. This is the
      exact case that must NEVER require `--allow-downgrade`: a legitimate TTL-driven or
      env-gated demotion to NOT-VERIFIED is not a "downgrade" in this guard's sense, only an
      explicit regression to `FAIL`/`PARTIAL` is.
  </behavior>
  <action>
Create `quality/runners/_persist_guard.py` exporting a function (e.g.
`committed_head_statuses(repo_root: Path, cat_path: Path) -> dict[str, str] | None`) that reads the
LAST COMMITTED version of a given catalog path via
`subprocess.run(["git", "show", f"HEAD:{relative_path}"], cwd=repo_root, capture_output=True,
text=True)`. If the subprocess exits non-zero (e.g. the path doesn't exist in HEAD â€” a brand-new
catalog file), return `None` (no baseline â€” NOT an error). Otherwise parse the stdout as JSON and
return `{row["id"]: row.get("status") for row in data.get("rows", [])}`.

Add a second function `refuse_downgrade_without_flag(committed: dict[str,str] | None, new_rows:
list[dict]) -> list[tuple[str,str,str]]` that: if `committed is None`, returns `[]` (nothing to
compare against). Otherwise, for each row in `new_rows`, if `row["id"]` is in `committed` AND
`committed[row["id"]]` is in `{"PASS", "WAIVED"}` AND `row.get("status")` is in
`{"FAIL", "PARTIAL"}` (an EXPLICIT regression â€” a verifier actually ran and produced a worse
grade), append `(row["id"], committed[row["id"]], row["status"])`. Return the violation list
(empty = no downgrade detected).

**SC2 regression-vs-TTL rule (exact semantics â€” non-negotiable, prevents a wave-5/6 deadlock):** the
guard fires ONLY on this explicit `{FAIL, PARTIAL}` transition from a committed `{PASS, WAIVED}`
baseline. A transition to `NOT-VERIFIED` is NEVER a violation, REGARDLESS of cause (freshness-TTL
expiry, missing-verifier demotion, env-gated skip, exit-75 NOT-VERIFIED convention, or any other
demotion channel) â€” `NOT-VERIFIED` is this project's designed-in "row went stale / couldn't be
graded" channel, not a regression, and rows minted through it (e.g. the freshness-invariants mints
in later waves of this very phase) must never need `--allow-downgrade` to persist. Do NOT attempt
to distinguish "legitimate TTL NOT-VERIFIED" from "other-cause NOT-VERIFIED" by inspecting
transient flags like `_stale` â€” those are popped before persistence (see `run.py`'s
`row.pop("_stale", None)` etc.) and are not reliably available at the guard's call site; the status
value alone (`NOT-VERIFIED` vs `FAIL`/`PARTIAL`) is the ONLY signal this guard consults, and that is
sufficient â€” it draws the line at "did a verifier actually run and produce a worse grade" (blocked)
vs "did the row become ungraded for any reason" (always allowed).

Wire into `run.py`'s `main()`: right before the existing `if args.persist: save_catalog(cat_path,
data)` line, call `committed = _persist_guard.committed_head_statuses(REPO_ROOT, cat_path)` then
`violations = _persist_guard.refuse_downgrade_without_flag(committed, data["rows"])`. If
`violations` is non-empty AND `not args.allow_downgrade`: for EACH violation print to stderr
`REFUSED to persist {row_id}: committed status was {old}, this run graded {new}. Pass
--allow-downgrade to override: python3 quality/runners/run.py --cadence {args.cadence} --persist
--allow-downgrade`; do NOT call `save_catalog` for this catalog file; append the catalog's name to
a new `blocked_downgrade_catalogs` list (parallel to the existing `pending_mint` list). If
`violations` is non-empty AND `args.allow_downgrade` IS set, persist as normal (unchanged behavior)
but STILL print one notice line per violation (e.g. `ALLOWED downgrade (--allow-downgrade):
{row_id} {old} -> {new}`) so the downgrade is never silent even when explicitly permitted. After
the main per-catalog loop, if `blocked_downgrade_catalogs` is non-empty, set
`exit_code = max(exit_code, 1)` before the final `return exit_code` â€” a blocked downgrade always
surfaces as a failing run, never silently swallowed into an otherwise-green exit.

Add `--allow-downgrade` to `_build_arg_parser()` (`action="store_true"`, default `False`, help text
explaining the refuse-by-default contract, mirroring the existing `--persist` flag's help-text
style).
