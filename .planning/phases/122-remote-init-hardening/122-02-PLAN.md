    - `resolve_import_parent()` returns Ok(None) ONLY when git spawned successfully AND
      exited 1 AND stdout is empty (genuine ref-absent first fetch → parentless seed).
    - It returns Err on: (a) `Command…output()` returning Err (spawn failure / git not on
      PATH), (b) a non-zero exit that is NOT 1 (e.g. 128 corrupt-repo / bad-revision),
      (c) exit 0 WITH empty stdout — anomalous: `git rev-parse --verify --quiet` on a success
      exit always prints the resolved oid, so an empty stdout on exit 0 is a broken-git /
      malformed-subprocess signal, NOT a benign absence; error loudly rather than silently
      degrade to the parentless overlay.
    - The caller (handle_import_batch, main.rs:422) propagates the Err through fail_push so
      the operator sees the RPX-0508 coded teaching, NOT a silent parentless overlay.
    - The `refs/reposix/origin/main` present case still returns Ok(Some(ImportParent{…})).
  </behavior>
  <action>
1. In `crates/reposix-core/src/codes.rs`: add `pub const HELPER_IMPORT_PARENT_RESOLVE: &str
   = "RPX-0508";` to `ids` (with a doc comment mirroring the RPX-0507 style) and ONE
   `ExplainEntry` to `REGISTRY` with a non-empty cause/fix/recovery. Cause: the helper could
   not resolve the client's `refs/reposix/origin/main` tracking tip because a `git rev-parse`
   subprocess FAILED for a non-absence reason (git not on PATH, or the caller repo is
   corrupt / not a git dir) — this is distinct from RPX-0507 (backend unreachable) and from
   the benign first-fetch ref-absence. Fix: ensure `git` is installed and on PATH and the
   caller directory is a valid git repository. Recovery: e.g. `git --version` and
   `git -C <repo> rev-parse --is-inside-work-tree`. Keep every field `&'static str`.

2. In `crates/reposix-remote/src/main.rs`: change the inner `rev_parse` closure to return
   `anyhow::Result<Option<String>>` (or a small enum) distinguishing the cases:
   - `Command…output()` Err → `Err(...)` (spawn failure).
   - status success WITH non-empty stdout → Ok(Some(trimmed)).
   - status success WITH empty stdout → `Err(...)` — anomalous: `git rev-parse --verify
     --quiet` on exit 0 always prints the resolved oid, so empty stdout is a broken-git /
     malformed-subprocess signal (≤1 extra branch), not a benign absence; fail loud via the
     same RPX-0508 teaching rather than silently returning Ok(None).
   - status not success: exit code == 1 → Ok(None) (ref absent, the `--verify --quiet`
     contract); any OTHER non-zero code → `Err(...)`.
   Lift `resolve_import_parent()` to `-> anyhow::Result<Option<ImportParent>>`. Build the
   Err via a `teach_coded(ids::HELPER_IMPORT_PARENT_RESOLVE, …)` teaching. Do NOT
