<action>
Determine the bump target. Two sources, in order of preference:

1. **`cargo search gix --limit 5`** — lists the latest published gix version
   from crates.io. Filter to the most recent NON-YANKED release. (The `cargo
   search` output marks yanks; cross-check via the crates.io web UI if
   ambiguous — `https://crates.io/crates/gix` lists yanked status per version.)
2. **`https://crates.io/api/v1/crates/gix/versions`** if cargo-search is
   inconclusive — JSON `yanked: true|false` field per version.

Same procedure for `gix-actor`. Note: `gix-actor` is a transitive dep of `gix`
itself (Cargo.lock:825 shows `gix → gix-actor`); when we bump `gix` the
matching `gix-actor` typically follows automatically. Confirm post-bump by
re-reading Cargo.lock.

Record the chosen version pair (e.g., `gix = 0.83.0`, `gix-actor = 0.41.0`) in a
local note. Cite this version in the commit message AND the CLAUDE.md update.

If the bump introduces a major-version-shaped change (e.g., 0.82 → 0.90), STOP
and append a SURPRISES-INTAKE.md entry — multi-version jumps invite conflated
debugging during the v0.13.0 milestone. The owner-ratified path is "next
non-yanked release"; if "next non-yanked" is multiple versions away, that is a
surprise worth flagging.
</action>

## Task 01-T03 — Run workspace gates SERIALLY; surface and fix breakage

<read_first>
- `CLAUDE.md` "Build memory budget (load-bearing — read before parallelizing)"
  — confirm: at most one cargo invocation at a time; prefer per-crate when
  scope allows; pre-push covers workspace.
- `Cargo.toml` lines 1-44 (members list) — to know which crates exist for
  per-crate fallback if memory pressure manifests.
</read_first>

<action>
Run the workspace gates in this order, **strictly serial**:

```bash
cargo check --workspace
```

If GREEN → next.
If RED → diagnose. Common failure modes for a gix bump:
- A removed/renamed method: read the gix CHANGELOG for the bumped range
  (https://github.com/GitoxideLabs/gitoxide/blob/main/CHANGELOG.md), find the
  call site in `crates/reposix-cache/` or `crates/reposix-remote/`, port it
  forward. Cite the CHANGELOG entry in the inline code comment for the fix.
- A removed feature flag: `crates/reposix-cache/Cargo.toml:24` requests
  `features = ["tree-editor"]`; confirm it still exists in the bumped gix.
- New required-dependency surface: rare for a single-minor bump but possible.

```bash
cargo clippy --workspace --all-targets -- -D warnings
```

If RED on a NEW lint that wasn't fired pre-bump, fix it inline (do NOT add
`#[allow(...)]` without a rationale comment per CLAUDE.md). New lints fired by
a gix bump are typically `clippy::needless_pass_by_value` or
`clippy::module_name_repetitions` — both require a one-line code change, not a
suppression.

```bash
cargo nextest run --workspace
```

If GREEN → done. If RED, the failing test names a specific gix call; fix the
call site, re-run nextest. If memory pressure manifests during nextest (the
symptom: test binaries OOM-killed before they finish; rare on this machine but
possible), fall back to per-crate:

```bash
for crate in reposix-core reposix-sim reposix-cache reposix-remote reposix-cli reposix-github reposix-confluence reposix-jira reposix-quality; do
  cargo nextest run -p "$crate"
done
```

Document the per-crate fallback in the commit body if used (it's an
Eager-resolution observation worth recording; not a defect).

Do NOT run `cargo check / clippy / nextest` in parallel. Do NOT spawn a
subagent that runs `cargo` while this task is running. CLAUDE.md "Build memory
budget" is non-negotiable; the VM has crashed twice from this.

If a test failure surfaces a gix-side bug NOT a reposix-side bug (rare but the
failure mode is "the bumped gix is also broken"), STOP, append to
SURPRISES-INTAKE.md (severity HIGH), and fall back to the prior version pinned
EXPLICITLY (e.g., the version BEFORE 0.82.0 — `gix = "=0.81.x"` if 0.81 was
non-yanked). The phase still ships HYGIENE-01 by closing the yanked-pin gap;
"latest non-yanked" is the goal but "any non-yanked" is the floor.
</action>
