<objective>
Wave 2 — execute the audit. Read `.planning/research/v0.11.1/repo-organization-gaps.md` rec-by-rec; classify each item; render the audit table at `quality/reports/audits/repo-org-gaps.md`; commit a verification script that asserts the audit is complete and consistent.

This is the load-bearing wave for ORG-01. The audit document IS the requirement-closure artifact. Wave 1 locked the contract (the row exists; the file path is committed); Wave 2 materializes the artifact.

Approach: based on STATE.md + git log, most of the rec #1 — rec #10 items are already CLOSED by prior milestones (v0.11.0 / v0.11.1 / v0.12.0 P56-P61). Wave 2's job is to AUDIT — verify the closure with `ls`/`git log`/`grep` for each item, attach citations, and produce the table.

Three deliverables:

1. **`quality/reports/audits/repo-org-gaps.md`** (NEW) — the audit table. Top-banner naming P62 + source doc + commit SHA at audit time. Body: markdown table with columns | # | Item (verbatim from source) | Disposition | Evidence/Citation |. Footer: list of items needing Wave 3 fix work.

2. **`scripts/check_repo_org_gaps.py`** (NEW) — the verifier. Reads both files. Asserts every numbered rec from the source is line-referenced in the audit table. Asserts every Disposition column value is one of: `closed-by-deletion | closed-by-existing-gate | closed-by-catalog-row | closed-by-relocation | closed-by-Wave-3-fix | waived | out-of-scope`. Exit 0 = PASS.

3. **Commit** with title `docs(p62): execute repo-org gap audit + commit verifier script`. Refs ORG-01.

**Why a script?** Per CLAUDE.md OP-4 ("Ad-hoc bash is a missing-tool signal"): "if the next agent needs to reproduce 'is the audit complete?', they need a committed script not a shell pipeline." The script is the audit's machine-checkable invariant. It runs as part of Wave 6's verifier subagent evidence.

**Why not put the script under `quality/gates/structure/`?** Because this script is one-shot per audit — it doesn't run on every pre-push. The catalog row that DOES run on every pre-push (`structure/repo-org-audit-artifact-present`) only checks the artifact's PRESENCE, not its consistency. The Wave 6 verifier dispatches the consistency check; in steady state the audit doesn't change row-by-row, so daily verification is wasteful. If a future audit re-opens the doc, the script gets re-run by hand or by the next P62-shaped phase.

**Pre-check evidence (Task 1):** before drafting the audit, run a survey shell to confirm the actual repo state. The README of the audit doc was 2026-04-26; STATE has moved a lot since. Items already-closed get classified as such with `ls`/`git log -1 --diff-filter=D` evidence.
