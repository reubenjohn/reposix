**Boundary analysis dimensions:**

| Question | scripts/catalog.py | quality/runners/verdict.py |
|---|---|---|
| What does it track? | files (paths) | gates (id slugs) |
| What is the unit? | one file's disposition | one verifier's outcome |
| What is the lifecycle? | per-session (this is what I'm working on) | per-cadence (this gate runs every pre-push) |
| What status set? | KEEP/TODO/DONE/REVIEW/DELETE/REFACTOR (work-state) | PASS/FAIL/PARTIAL/WAIVED/NOT-VERIFIED (gate-state) |
| What's the source artifact? | per-file rows in `.planning/v0.11.1-catalog.json` | per-row verifier artifacts in `quality/reports/verifications/<dim>/<slug>.json` |
| Who sets status? | the developer (via `catalog.py set`) | the verifier subprocess (auto-set by runner) |
| What's the cadence? | session-bounded (one v0.11.x catalog file per session) | infinite (gates run forever; rows persist across sessions) |
| What's the merge criteria? | "all TODO files closed → milestone ships" | "all P0+P1 rows GREEN → CI passes" |

**The two systems answer different questions** — one is "what files do I touch in this session?" (planner/IDE state); the other is "what gates are GREEN this run?" (CI/verifier state). Conflating them would degrade both:
- A gate's PASS doesn't say anything about whether the file behind it is "DONE" in the v0.11.x sense (a gate can PASS while its source is still REVIEW-pending).
- A file's DONE doesn't say anything about whether its associated gate stays GREEN over time.

**The shared infrastructure** that COULD merge them in v0.13.x:
- Both are JSON-on-disk catalogs read by stdlib Python.
- Both have a `subprocess.run`-driven mutation lifecycle.
- Both have markdown-rendering logic.

The shared abstraction would be a `Catalog<T>` class where `T` is either `FileDisposition` or `GateOutcome`. **Out of scope for v0.12.0** per Wave A's boundary doc.
