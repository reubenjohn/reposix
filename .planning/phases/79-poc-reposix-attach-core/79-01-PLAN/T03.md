Create `research/v0.13.0-dvcs/poc/path-b.sh`. This script simulates the
SoT-first sequencing without a real bus URL parser — the goal is to surface
findings about ordering, audit-row writes, and what mirror-lag observability
needs.

Strategy: spin up a SECOND simulator instance to play the "GH mirror" role
(the simulator's bare-repo cache writes are the closest local analog to
"git push to mirror"). The POC manually orchestrates:

1. Start simulator-A on `POC_SIM_PORT` (the SoT). Seed with record id=1.
2. Start simulator-B on `POC_SIM_PORT+1` (the mirror surrogate). Empty.
3. Manually invoke a write: `curl -X PATCH http://localhost:${POC_SIM_PORT}/projects/demo/records/1 -d '{"body_md":"updated by POC"}'` — succeeds (SoT write).
4. Manually attempt to "push to mirror" by writing the same patch to simulator-B. SIMULATE FAILURE: kill simulator-B BEFORE the second write, so the mirror-write fails with a connection error.
5. Observe + log: SoT update succeeded; mirror update failed; "lag" is now > 0 (we can express this as "mirror has version 0; SoT has version 2" since the simulator increments versions on PATCH).
6. Restart simulator-B; replay the patch successfully. Observe: lag is now 0.

This is NOT the production bus algorithm; it's a STAGED reproduction that
exercises the sequencing decisions the bus algorithm depends on. The
findings the POC must surface:

- Does the sim's `audit_events` table record both writes-attempt + the
  failure? (Lookup via the sim's audit endpoint or by inspecting the sim's
  DB.) If audit logging requires a custom hook the production bus will
  need, that's a FINDING tagged INFO.
- What does `last_fetched_at` actually mean in the cache state, and does it
  need bumping after a mirror-write failure? (Per architecture-sketch step
  6: SoT write success → update last_fetched_at. So lag tracking lives
  outside last_fetched_at; it's a separate ref state.) Findings about how
  to express lag without confusing the existing cache state model are
  tagged INFO or REVISE.
- Are there any ordering interleavings where a partial write leaves the
  cache in an unrecoverable state? Specifically: if SoT-write succeeds,
  cache audit row is written, then the helper crashes BEFORE the mirror
  write is attempted, what should the next push do? The POC should EXPLICITLY
  test this scenario (e.g., kill the path-b.sh script between SoT-write and
  mirror-write).

```bash
#!/usr/bin/env bash
# POC path (b): bus SoT-first observing mirror lag.
set -euo pipefail
SCRIPT_DIR="$(cd -- "$(dirname -- "${BASH_SOURCE[0]}")" &> /dev/null && pwd)"
cd "${SCRIPT_DIR}"

echo "=== Path (b): bus SoT-first observing mirror lag ==="

# Two sims: SoT (A) on POC_SIM_PORT, mirror (B) on POC_SIM_PORT+1.
SIM_A_PORT="${SIM_PORT}"
SIM_B_PORT=$((SIM_PORT + 1))

# Start sim A (SoT).
cargo run -p reposix-sim --quiet -- --bind "127.0.0.1:${SIM_A_PORT}" --ephemeral --no-seed &
SIM_A_PID=$!

# Start sim B (mirror). Different ephemeral DB.
cargo run -p reposix-sim --quiet -- --bind "127.0.0.1:${SIM_B_PORT}" --ephemeral --no-seed &
SIM_B_PID=$!

sleep 2  # both bind

# Seed both with record id=1, version=1.
curl -fsS -X POST "http://127.0.0.1:${SIM_A_PORT}/projects/demo/records" \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"title":"hello","state":"open","body_md":"v1"}'
curl -fsS -X POST "http://127.0.0.1:${SIM_B_PORT}/projects/demo/records" \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"title":"hello","state":"open","body_md":"v1"}'

# Step (1) SoT-first write: PATCH sim A.
echo "[POC b] Step 1: SoT write to sim A (port ${SIM_A_PORT})"
curl -fsS -X PATCH "http://127.0.0.1:${SIM_A_PORT}/projects/demo/records/1" \
  -H 'Content-Type: application/json' \
  -d '{"body_md":"v2 from POC"}' || echo "[POC b] SoT write FAILED (unexpected)"

# Step (2) Simulate mirror-write FAILURE: kill sim B before patch.
echo "[POC b] Step 2: kill sim B (mirror) to simulate mirror-write failure"
kill "${SIM_B_PID}" 2>/dev/null || true
sleep 1

# Step (3) Try the mirror write; expect connection refused.
echo "[POC b] Step 3: mirror write attempt (expect failure)"
if curl -fsS -m 2 -X PATCH "http://127.0.0.1:${SIM_B_PORT}/projects/demo/records/1" \
   -H 'Content-Type: application/json' \
   -d '{"body_md":"v2 from POC"}'; then
  echo "[POC b] UNEXPECTED: mirror write succeeded"
else
  echo "[POC b] EXPECTED: mirror write failed (connection refused). Lag is now > 0."
fi

# Step (4) Restart sim B; replay; observe lag goes to 0.
cargo run -p reposix-sim --quiet -- --bind "127.0.0.1:${SIM_B_PORT}" --ephemeral --no-seed &
SIM_B_PID=$!
sleep 2
curl -fsS -X POST "http://127.0.0.1:${SIM_B_PORT}/projects/demo/records" \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"title":"hello","state":"open","body_md":"v1"}' || true
echo "[POC b] Step 4: replay mirror write"
curl -fsS -X PATCH "http://127.0.0.1:${SIM_B_PORT}/projects/demo/records/1" \
  -H 'Content-Type: application/json' \
  -d '{"body_md":"v2 from POC"}'
echo "[POC b] EXPECTED: mirror caught up. Lag now 0."

# Step (5) Inspect audit trail (sim's audit endpoint).
echo "[POC b] Step 5: SoT audit"
curl -fsS "http://127.0.0.1:${SIM_A_PORT}/projects/demo/audit" || echo "[POC b] audit endpoint not implemented; finding"
echo "[POC b] Step 6: Mirror audit"
curl -fsS "http://127.0.0.1:${SIM_B_PORT}/projects/demo/audit" || echo "[POC b] audit endpoint not implemented; finding"

# Cleanup local
kill "${SIM_A_PID}" "${SIM_B_PID}" 2>/dev/null || true
```
