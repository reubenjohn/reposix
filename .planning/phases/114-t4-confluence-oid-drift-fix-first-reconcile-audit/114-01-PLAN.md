<objective>
Fix the Confluence `list_records`-vs-`get_record` oid-drift product defect at its root
cause (FIX-01): `list_issues_impl` requests the space's pages with NO `body-format` query
param, so Confluence returns every page's body EMPTY, `build_from` computes the tree-blob
oid from that empty-body render, and `read_blob` later hard-aborts with `Error::OidDrift`
because `get_record` (which DOES request `?body-format=atlas_doc_format`) returns the REAL
content. The mismatch is deterministic and structural — not the eventual-consistency race
the code comment assumes (verified: 114-RESEARCH.md § Root Cause).

Fix = render-parity in the adapter (Success Criterion 3, Option 1): add
`&body-format=atlas_doc_format` to the LIST url so the list body and the get body decode
through the SAME `translate()` to byte-identical output for every ADF-native page. This is
a single-query-param, zero-new-code-path change; it does NOT touch `read_blob`'s drift
check (that guard is working as designed — Pitfall 1) and does NOT add per-page GETs to the
list walk (that would defeat the lazy-blob/ADR-010 architecture — Pitfall 2 / Option 2).

Purpose: Confluence page checkouts stop aborting; the P0 real-backend gate goes GREEN.
Output: adapter fix + a render-parity contract test + two stale-doc-comment corrections.
</objective>
