<objective>
Audit `reposix sync --reconcile`'s oid-drift recovery claim (FIX-02, GTH-V15-19) and
correct the doc comments to their real scope. `--reconcile` is NOT a separate code path —
`sync.rs::run` calls `Cache::build_from()` directly (the exact same `list_records` walk +
render/hash as the seed). So it CAN heal tree↔`oid_map` coherence drift (ghost/missing
rows) and genuine eventual-consistency races, but it CANNOT heal the systematic
list-vs-get representation-drift class Phase 114 targets: a second `build_from` recomputes
the SAME empty-body-derived oid deterministically (verified: 114-RESEARCH.md § FIX-02).

Two deliverables:
1. A backend-agnostic `oid_drift_reconcile.rs` test (modeled on `pagination_prune_safety.rs`'s
   `CappingMock`) that PROVES, against a reproduction rather than an assumption: divergent
   list/get bodies → `Error::OidDrift`; a repeated `build_from` leaves the stale list-oid
   unchanged; aligned bodies resolve cleanly.
2. Corrected doc comments in `error.rs` (`Error::OidDrift`), `sync.rs` (module + `run` fn),
   and `main.rs` (the `Sync` clap doc, for consistency) that name all three coexisting drift
   classes precisely — NOT over-corrected to "reconcile never recovers oid-drift" (Pitfall 3).

Note on SC4's literal wording: SC4 says "builder.rs/cache.rs doc comments," but the actual
claim-bearing comments are in error.rs + sync.rs + main.rs (114-RESEARCH.md § FIX-02, table
+ Assumption A3). `cache.rs:575-582` (`write_last_fetched_at`) is ACCURATE for cursor drift
and needs NO edit — only a read-and-confirm. This research grep-verified table is
authoritative over SC4's literal file names.
