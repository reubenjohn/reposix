## Decision log (in conversation order)

### D1 — Yes, build a coherent Quality Gates system (not bespoke checks)

**Owner question:** "Can you think about how to organize the checks to be simpler and more effective? e.g. end-state.py is a weird way to name the check that validates releases? Maybe we want to call it release gate?"

**Decision:** Build a dimension-tagged Quality Gates system (`quality/gates/<dim>/`) that REPLACES ad-hoc scripts. The mental model: dimension × cadence × kind. The naming critique is real — `scripts/end-state.py` does five different jobs that don't share a meaningful umbrella beyond "they happen at session end."

**Rationale:** every quality miss in v0.11.x produced a bespoke script. The pattern repeats. The structural break is to give every regression class a home, every check a verifier, every verifier an artifact, every artifact a subagent grader.

### D2 — Use "gates" as the umbrella term

**Owner answer to "Naming — Quality Gates / `quality/` as the umbrella, or different word?":** "1. gates sounds good."

**Decision:** `quality/gates/<dim>/`. The word "gate" implies blocking, named, owned.

### D3 — `end-state.py` becomes a thin shim with anti-bloat comments

**Owner answer to "End-state.py keep or rename?":** "2. Yes make it thin. But maybe leave comments - my concern is that an agent will bloat it instead of investing into strategic gates."

**Decision:** `scripts/end-state.py` reduced to ≤30-line shim that delegates to `quality/runners/verdict.py session-end`. Anti-bloat header comment explicitly tells future agents: "this file does not grow; new gates go under `quality/gates/<dim>/`." Captured as STRUCT-02 + SIMPLIFY-02.

**Rationale:** owner has lived through enough sessions to know that agents take the path of least resistance. Without the anti-bloat warning at the top of the file, the next agent will add one more row to `end-state.py` instead of building the gate in the right place.

### D4 — Parallel migration; pivot to hard-cut if it stalls

**Owner answer to "Migration boldness — parallel or hard cut?":** "3. Sounds good but if in practice its stopping progress, pivot."

**Decision:** Ship new framework alongside old. Compare verdicts in `quality/reports/verdicts/parity/` for two pre-push cycles. Hard-cut (delete old) only after parity is demonstrated. Pivot to hard-cut earlier if parallel migration stalls progress.

**Rationale:** OP-5 reversibility — keep both running until the new system proves itself. But don't ship a half-migrated system forever.

### D5 — Repo-org-gaps becomes a phase of v0.12.0 (P62)

**Owner answer to "Subjective gates ambition — start small or seed with everything in repo-org-gaps.md?":** "4. I just don't want to loose track of repo-org-gaps can you make it a later phase of this same milestone?"

**Decision:** P62 is a dedicated phase that audits `.planning/research/v0.11.1-repo-organization-gaps.md` against current state. Each remaining gap → either fix + add a structure-dimension catalog row that prevents recurrence, OR file a waiver with reason. ORG-01 in REQUIREMENTS.

**Rationale:** the gaps doc is real work that would otherwise rot. Folding it into v0.12.0 means each closure becomes a permanent catalog row instead of a one-off fix.

### D6 — Container rehearsal cadence is post-release, not weekly

**Owner question:** "5. I don't understand, doesn't it only run when we make a release?"

**Decision:** Container rehearsal of install paths is `cadence: post-release` — runs once per release after assets ship, NOT weekly. The cheap weekly check is just `HEAD https://.../reposix-installer.sh → expect 200 + >1KB + #! prefix`.

**Rationale:** the agent had originally planned nightly container rehearsal — too expensive for what it catches. Owner correctly observed: container rehearsal proves a fresh user can install from the LATEST release, which only changes when we release. So tie the gate to the release lifecycle, not a clock.

### D7 — Cadence is "weekly", not "nightly"

**Owner message (mid-discussion):** "Instead of "nightly" lets make it weekly."

**Decision:** Replace every "nightly" reference with "weekly" throughout the milestone plan. The cron drift detectors (asset-exists URL HEADs) run weekly.

**Rationale:** cost-conscious. Weekly catches the regression class within 7 days, which is fine — install URLs don't go bad on the hour. Nightly = 7× more cost for marginal latency improvement.

### D8 — Install fix is the FIRST phase of v0.12.0 (P56), not a separate /gsd-quick

**Owner answer to "Install fix urgency — /gsd-quick outside the milestone or first phase of v0.12.0?":** "6. First phase. I plan to run the whole milestone soon autonomously with subagents without me in the loop so its very important the plan is robust and that it follows the operating principles and that it accomodates feedback and pivoting as there are always surprises as long as we don't settle for a poorer end state. If there are important ideas or innovations, go for it rather than surfacing it to me."

**Decision:** P56 ships the install fix as the entry-point phase of v0.12.0 (not a separate `/gsd-quick`). The whole milestone runs autonomously.

**Rationale:** cleaner narrative. The install fix is the proximate cause of the milestone — it belongs IN the milestone. Plus the autonomous mode grant means the agent should innovate proactively rather than ask permission.

**Key mandate from this decision:** the plan must be robust against autonomous failure modes (drift, half-done, silent downgrade, lost context, bloat, cascading break). Captured in `v0.12.0-autonomous-execution-protocol.md`.

### D9 — CLAUDE.md updated PER PHASE, not just at the end

**Owner message (mid-execution):** "I think we need to update the CLAUDE.md in each phase not just at the end?"

**Decision:** Mandatory CLAUDE.md update per phase (QG-07). Each phase that introduces a new file, convention, gate, or operational rule MUST update the relevant CLAUDE.md section in the SAME PR. The verifier subagent grades this as a phase-close requirement.

**Rationale:** the original plan had CLAUDE.md only updated in P63 (final phase). That would mean the autonomous agent picking up P58 wouldn't have CLAUDE.md mention the QG protocol because we'd not have updated it yet — every later phase ungrounded. Same recurrence pattern as the §0 misses in v0.11.x. Per-phase update prevents it.

**Anti-bloat sub-rule:** each phase appends a paragraph + code reference; deletions are encouraged when superseded; no full rewrites.

### D10 — Aggressive simplification: absorb existing scripts/examples into the framework

**Owner message (mid-execution):** "My hope is that this new system can greatly simplify the codebase. There are a lot of scripts, examples/, etc lying around. I am not saying that they are not valuable, but I would aggressively look for opportunities to simplify if they can be swallowed by this framework as by incorporating them into the various phases/plans of this milestone."

**Decision:** Added SIMPLIFY-01..12 requirements covering every existing `scripts/*` and `examples/*` surface. Each names the source surface, target home in `quality/gates/<dim>/`, and the phase that absorbs it. After v0.12.0, `scripts/` holds only `hooks/` and `install-hooks.sh`.

**Rationale:** the framework should REPLACE the ad-hoc surfaces, not add a new layer on top. SIMPLIFY-12 (P63 final audit) enforces: `find scripts/ -maxdepth 1 -type f | grep -v hooks | grep -v install-hooks.sh` returns empty (or every remaining file has an explicit waiver in `quality/catalogs/orphan-scripts.json`).

**Examples-specific:** every `examples/0[1-5]-*/run.sh` becomes a docs-repro catalog row (container-rehearsal-kind, post-release cadence). The `examples/` README documents that each example is a tracked gate.

### D11 — Top-level REQUIREMENTS.md and ROADMAP.md split per-milestone

**Owner question:** "Is the requirements doc getting too monolithic?"

**Decision:** Extract historical REQUIREMENTS sections into per-milestone files inside `*-phases/`:
- `.planning/milestones/v0.11.0-phases/REQUIREMENTS.md` — v0.11.0 + v0.11.1 sections
- `.planning/milestones/v0.10.0-phases/REQUIREMENTS.md` — v0.10.0 section
- `.planning/milestones/v0.9.0-phases/REQUIREMENTS.md` — v0.9.0 + v1 + Future + Out of Scope + Traceability sections

Top-level `.planning/REQUIREMENTS.md` becomes v0.12.0 active + a "Previously validated" index with relative links. Same rule for ROADMAP.md applies in spirit (top-level stays slim).

**Captured as catalog row:** QG-08 in REQUIREMENTS — top-level `.planning/REQUIREMENTS.md` MUST contain ONLY the active milestone + index. STRUCT catalog row enforces the convention going forward.

**Rationale:** per CLAUDE.md §0.5 the convention already exists (per-milestone REQUIREMENTS.md inside `*-phases/`), but the historical sections never moved — they accumulated in the top-level file. The freshness invariant `freshness/no-loose-roadmap-or-requirements` doesn't catch THIS case because top-level `.planning/REQUIREMENTS.md` is allowed (the invariant only blocks files at wrong paths). v0.12.0 adds a structure-dimension catalog row that enforces SCOPE — top-level is index + active only.

## Owner preferences carried over from earlier sessions (unchanged)

These are durable constraints; the autonomous agent should follow them without re-asking:

- **No walkthrough / morning-brief / session-recap docs.** Operational handovers are anti-patterns; planning artifacts (REQUIREMENTS.md, ROADMAP.md, STATE.md, research/) are the source of truth.
- **Subagent delegation aggressive.** Coordinator should not type code a subagent could type. Per CLAUDE.md OP-2 + global guidance.
- **Push frequently.** Pre-push hook gates fmt + clippy + check-docs-site + banned-words + (now) `quality/runners/run.py --cadence pre-push`.
- **One cargo invocation at a time.** RAM budget per CLAUDE.md "Build memory budget" section. VM has crashed twice from parallel cargo.
- **No skills changes without explicit owner approval.** `.claude/skills/` is owner-approval-gated. (Note: `reposix-quality-review` skill in P61 is pre-approved as part of v0.12.0 scope.)
- **Owner is reubenjohn (`reubenvjohn@gmail.com`).** gh CLI is authenticated. Repo secrets settable via `gh secret set`.
- **Banned word: `replace`.** Use `migrate to` / `for the 80%` / `rewrite as` / `complement` / `alongside`. The pre-push hook enforces it.
- **Never delegate `gh pr checkout` to a bash subagent without isolation.** Use a worktree (`git worktree add /tmp/pr-N`) or `/tmp/<branch>-checkout`. (CLAUDE.md "Subagent delegation rules" lesson learned at commit `5a91ae2`.)
