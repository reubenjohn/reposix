# v0.12.0 — Open Questions and Deferrals

> **Audience.** The agent executing v0.12.0 + the agent picking up v0.12.1. This doc lists what's EXPLICITLY out of v0.12.0 scope and why, plus open questions where v0.12.0 makes a judgment call but the next milestone may revisit.

## Out of v0.12.0 scope (deferred to v0.12.1)

### Perf dimension (full)

**v0.12.0 ships:** stub. SIMPLIFY-11 moves `scripts/bench_token_economy.py`, `scripts/test_bench_token_economy.py`, `scripts/latency-bench.sh`, `benchmarks/fixtures/*` into `quality/gates/perf/` with a `perf-targets.json` catalog stub.

**v0.12.0 does NOT ship:** the gate logic that cross-checks bench output against headline copy. (E.g., a check that says "the latency.md p50 cell value is within ±2ms of the `8 ms` headline in docs/index.md").

**Why deferred:** the cross-check requires deciding what the soft thresholds are for each headline number, and what tolerance is acceptable for natural variation. That's a meaningful design conversation that doesn't need to happen in v0.12.0 — the framework lands first, then v0.12.1 fills in the perf rubric.

**Captured in:** MIGRATE-03 (file the v0.12.1 carry-forward).

### Security dimension (full)

**v0.12.0 ships:** thin home (`quality/gates/security/` exists) but no actual gates.

**v0.12.0 does NOT ship:**
- Allowlist enforcement gate (a check that no new `reqwest::Client::new()` site exists outside the factory — currently enforced by clippy `disallowed_methods`, but not via Quality Gates).
- Audit immutability test (a check that the SQLite triggers blocking UPDATE/DELETE on audit tables haven't regressed).
- Cosign / SBOM / SLSA / Apple notarization / Authenticode supply-chain hardening (carry-forward from v0.11.1 friction row 19).

**Why deferred:** clippy already enforces the allowlist; security gates are valuable as defense-in-depth but not user-facing-urgent. The supply-chain hardening is a separate body of work that warrants its own milestone discussion.

**Captured in:** MIGRATE-03.

### Cross-platform container rehearsals

**v0.12.0 ships:** ubuntu:24.04 baseline only.

**v0.12.0 does NOT ship:** windows-2022 or macos-14 GH-runner-based container rehearsals.

**Why deferred:** windows + mac runners are real money ($0.08/min for windows, $0.16/min for mac on GH Actions). Per-release cost would add up. Owner has approved deferring to v0.12.1 with "I'd start small."

**Captured in:** MIGRATE-03.

### `Error::Other` 156→144 partial migration completion

**v0.12.0 ships:** nothing on this front. v0.11.1 POLISH2-09 partial-shipped (3 typed Error variants + sim.rs migrated; ~150 backend sites pending). The carry-forward from v0.11.1 is preserved as v0.12.1 work.

**Why deferred:** typed-error migration is unrelated to Quality Gates scope. Folding it into v0.12.0 would dilute the milestone's focus. The Quality Gates framework, once shipped, will give the v0.12.1 agent a place to put a gate that PREVENTS new `Error::Other(String)` sites from being introduced.

**Captured in:** MIGRATE-03.

### Threat-model rewrite for v0.9.0 architecture

**v0.12.0 ships:** nothing. The FUSE-era threat model at `.planning/research/v0.1-fuse-era/threat-model-and-critique.md` stays as-is.

## Open questions where v0.12.0 makes a judgment call

### Q1 — Does `scripts/catalog.py` overlap with `quality/runners/verdict.py`?

**Status:** SIMPLIFY-03 audits this in P57.

**Initial assessment:** different domains. `scripts/catalog.py` tracks per-FILE planning catalog (which files are touched in this session); `quality/runners/verdict.py` tracks per-CHECK quality verdict. Keep separate; document the boundary in `quality/catalogs/README.md`.

**Reopen if:** P57's audit reveals significant overlap.

### Q2 — Should subjective rubric grading be deterministic or accept variation?

**Status:** P61 ships hero-clarity, install-positioning, headline-numbers as subjective rubrics with subagent grading.

**Initial assessment:** accept variation; require numeric scores with a stability check (re-grade twice; flag if delta > N).

**Reopen if:** subagent grading proves wildly inconsistent (same artifact graded 8/10 then 5/10 then 9/10).

**Pivot rule (in PROTOCOL.md):** if instability > tolerance, switch rubric to a more deterministic format (e.g., explicit assertion checklist instead of holistic score).

### Q3 — Does `quality-post-release.yml` chain reliably?

**Status:** P58 ships this workflow as a separate file triggered by `release.yml` success.

**Initial assessment:** GitHub Actions `workflow_run` triggers can be brittle in subtle ways (race conditions, run-vs-trigger ambiguity).

**Reopen if:** P58's first deployment shows brittleness.

**Pivot rule:** fall back to a single workflow with `cron + workflow_dispatch + workflow_run` triggers.

### Q4 — Does the runner's discovery scale?

**Status:** P57 ships `quality/runners/run.py` reading every catalog file per invocation.

**Initial assessment:** N (catalogs) is small (≤10 files), M (rows per catalog) is moderate (≤50 rows), K (cadences) is small (6). O(N×M) per cadence per invocation. ~few hundred ms.

**Reopen if:** pre-push budget (60s) gets squeezed by runner overhead.

**Pivot rule:** add `quality/runners/_cache.py` to cache parsed catalogs per run + skip rows where `last_verified` is fresher than the catalog file's mtime.

### Q5 — How does `examples/03-claude-code-skill/` fit the docs-repro pattern?

**Status:** unlike the other 4 examples (bash/python/bash/bash), example 03 is a Markdown spec for a Claude Code skill that humans invoke from Claude Code. Container rehearsal doesn't apply.

**Initial assessment:** mark as `kind: manual` in the catalog with `freshness_ttl: 90d`. Owner-on-demand rerun.

**Reopen if:** the v0.12.1 expansion to mac/windows rehearsals introduces a pattern for skill-based examples.

## Items the autonomous agent should escalate (NOT decide alone)

If the agent encounters one of these, file a SURPRISES.md entry + STATE.md cursor update + STOP:

- **Owner credentials needed for real-backend testing.** Don't try to reproduce real-backend issues without `JIRA_EMAIL` / `JIRA_API_TOKEN` / `ATLASSIAN_API_KEY` etc.
- **Skill changes** outside `reposix-quality-review` (which is pre-approved for P61). `.claude/skills/` is owner-approval-gated.
- **Force-pushes to main.** Always escalate.
- **Workflow permissions changes** (default_workflow_permissions, secret rotation).
- **Closing GitHub issues / PRs** that aren't owned by the milestone.

For everything else, the autonomous-mode grant from D8 applies: "If there are important ideas or innovations, go for it rather than surfacing it to me."
