### GTH-V15-67 — `cred-hygiene.sh` has no inline test-fixture allow-marker; security-regression tests can't use realistic provider-prefixed fixtures
- **Source:** NOTICED during P120 CLOSE while unblocking the phase push: the WR-01 credential-redaction regression test (`bus_handler.rs::tests::wr01_mirror_partial_fail_scrubs_token_from_both_audit_row_and_diag`) used a `ghp_`-shaped fake token (a `ghp_` prefix followed by 20+ alphanumerics — literal elided so this GTH entry does not itself trip the scanner) as its SECRET fixture, which collided with `cred-hygiene.sh`'s own `ghp_[A-Za-z0-9]{20,}` P0 pre-push pattern — the redaction test proving a leak is FIXED tripped the scanner that exists to catch leaks. · **Severity: LOW-MED (recurrence risk: every future redaction/security-regression test that wants a REALISTIC provider-prefixed fixture hits the same wall)** · STATUS: OPEN — tag quality-gates / security / cred-hygiene.
- **What:** `cred-hygiene.sh` (`quality/gates/structure/cred-hygiene.sh`) greps the outgoing diff for provider-shaped secret patterns with NO per-line escape hatch — unlike `teach_scan.py`'s `// teach-exempt: ok` or `test-name-vs-asserts.sh`'s `// test-name-honesty: ok` markers, there is no way to say "this literal is an intentional test fixture, not a leaked credential" short of choosing a fixture value that structurally avoids every pattern (the P120 fix: renamed to `"mirror-pushtoken-REDACTME-abc123def456"`, a non-prefixed value — valid here because `redact_userinfo` strips URL userinfo STRUCTURALLY, format-agnostic to the token's shape, so a non-prefixed fixture exercises the identical code path). That workaround is NOT always available — a test asserting specifically that a REAL provider's token SHAPE (e.g. `ghp_`, `AKIA`, `sk-proj-`) is redacted needs the real shape to be a faithful regression guard, and today has no sanctioned way to keep it without either tripping the P0 gate or adding the fixture to an EXCLUDE_DIRS-style bypass (too broad — would blind the gate for the whole file/dir, not just the one fixture line).
- **Fix-sketch:** add an inline allow-marker the scan loop checks per-line before flagging a hit — e.g. a trailing `// cred-hygiene:allow-fixture` (or `# cred-hygiene:allow-fixture` for non-Rust) comment on the same line as the matched literal. `cred-hygiene.sh`'s `content` variable is already read as full-file text; the pattern loop would need to become line-aware (currently `grep -qE` against the whole blob) to check for a trailing marker on the SPECIFIC matched line, not just presence anywhere in the file. This is a **P0 security-gate bypass mechanism** and needs careful review before landing — scope it tightly (marker must be on the exact matched line, never file-wide; consider requiring a `<reason>` suffix like the other two markers; consider whether the marker itself should be excluded from `EXCLUDE_DIRS`-style scanning so a marker can't be used to launder a real leak). Do NOT implement as a quick fix — this is deliberate future work, flagged here rather than built now.
- **Effort:** small-medium — the line-aware rewrite of the grep loop plus the marker-recognition logic is small; the review/threat-model sign-off for a P0 gate bypass mechanism is the larger, non-mechanical cost.

### GTH-V15-70 — CLI fetch-FAILURE tests that run against the default `127.0.0.1:7878` are silently sim-port-dependent (false-RED if a sim is up)
- **Source:** NOTICED during P121 W3.6 (transport-test isolation audit), triaged + filed in P121 W4. · **Severity: LOW (CI has no simulator listening on that port, so the hazard is dormant in CI; it bites a local dev who leaves `reposix sim` running)** · STATUS: OPEN — tag agent-ux / test-isolation.
- **What:** any CLI integration test that drives `reposix init sim::demo <path>` (or a similar sim-backed command) WITHOUT overriding the origin, and asserts a NON-ZERO exit because the initial `git fetch` should fail, silently depends on nothing listening at the default `127.0.0.1:7878`. If a simulator happens to be up on that port (a dev left `reposix sim` running, or a parallel test bound it), the fetch SUCCEEDS, init exits 0, and the failure assertion (`assert!(!status.success())` / `expect_err`) panics → a false RED that has nothing to do with the code under test. **Verify-against-reality note (W4):** the specific test name carried in the W3.6 handoff (`init_errors_nonzero_when_initial_fetch_fails`) was NOT found verbatim in the current tree — the hazard is the CLASS, not one named test. The good-pattern counter-examples already in the tree isolate correctly: `agent_flow.rs`'s init success test sets `REPOSIX_SIM_ORIGIN` to its own ephemeral sim, and `no_truncate.rs::no_truncate_flag_exits_nonzero_when_backend_unreachable` uses a wiremock server on an OS-assigned port — neither trusts the well-known `:7878`. The risk is that a future (or as-yet-unaudited) failure-path test omits that override and regresses to the default-port assumption.
- **Fix-sketch:** for any failure-path test, bind an ephemeral port, immediately close it, and point the command at that now-guaranteed-dead port (via `REPOSIX_SIM_ORIGIN` / the relevant origin env var) instead of relying on `:7878` being free — or add a small shared test helper (`dead_origin()`) that returns such a URL so the pattern is one call, not copy-pasted per test. A lightweight audit sweep of the CLI test corpus for `reposix init`/`list` invocations lacking an explicit origin override would surface any current offenders.
- **Effort:** small (per-test env override, or one shared helper + a grep-driven sweep).
