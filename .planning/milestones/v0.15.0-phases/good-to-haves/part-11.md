### GTH-V15-99 — decouple the F-K4b grade-time demote from implicit `minted_at` activation (make it an explicit opt-in flag)
- **Source:** NOTICED at P126 (the `ba13553f` F-K4b grade-time-demote CI regression + its `quality/CLAUDE.md` write-up). Adding a write-once `minted_at` to a legacy `kind: mechanical` row that emits `asserts_passed` does TWO things at once: it defuses the `validate_row` load-time crash (the intended W1 fix) AND it silently activates `apply_pass_gates`'s F-K4b `asserts_congruent` grade-time check (gated on `if row.get("minted_at")`). The second activation is an invisible side effect of the first — `agent-ux/real-git-push-e2e` FAILed CI run 29687639465 at 0.68s the instant `minted_at` landed, because an `expected.asserts` entry described a mutually-exclusive `git < 2.34 → NOT-VERIFIED` branch the PASS run never executes. The coupling is documented in `quality/CLAUDE.md` § "minted_at also arms the GRADE-TIME F-K4b congruence gate", but no code-change row exists to actually decouple it. · **Severity: MEDIUM (correctness/footgun — a load-crash fix silently arms a second, differently-scoped grade gate; the next legacy-row `minted_at` add can repeat the demote)** · STATUS: OPEN — tag code / quality-runner (`quality/runners/_audit_field.py`).
- **What:** the failure mode is a genuine footgun: an agent adding `minted_at` to defuse the load crash (the well-documented W1 pattern) has no local signal that they've ALSO just armed F-K4b grade-time congruence on that row — and F-K4b is cadence-scoped differently (`real-git-push-e2e` is pre-pr/pre-release/on-demand, NOT pre-push), so the demote surfaces only in CI, not the local pre-push battery. The two concerns (audit-cutoff anchor vs. PASS-path congruence) ride the same field.
- **Fix-sketch:** split the two triggers. Give F-K4b its own EXPLICIT opt-in — e.g. a `asserts_congruence: true` (or `fk4b: true`) row field that `apply_pass_gates` checks INSTEAD of `if row.get("minted_at")` — so adding `minted_at` for the audit-cutoff/load-crash reason no longer implicitly arms grade-time congruence. Default the new flag OFF; opt-in only rows that genuinely want PASS-path congruence checking. Migrate the rows that currently RELY on the implicit coupling in the same change. Keep the existing `test_audit_field.py::TestFK4bMutuallyExclusiveBranch` regression green, add a test that `minted_at` WITHOUT the new flag does NOT demote.
- **Effort:** small-medium (~1-1.5h) — one field + one `apply_pass_gates` branch swap + a migration sweep of currently-coupled rows + 1-2 selftests. Verify no F-K4b-dependent row silently loses coverage in the migration.

### GTH-V15-100 — reconsider cadence membership of the 2 live-network badge P2s (~127s serial pre-pr cost)
- **Source:** NOTICED at P126 close (gate-sweep timing observation). `docs-build/badges-resolve` and `docs-build/p94-badges-real-vs-transient` are both live-network P2 rows that together add **~127s of serial cost to every pre-pr run** — they make real HTTP round-trips to badge providers (shields.io / CI-status / codecov endpoints) on every PR, which is a large fixed tax for a low-blast-radius (P2) freshness check that rarely changes verdict between PRs. · **Severity: LOW (CI cost / cadence-fit — non-blocking; the checks are correct, just expensively scheduled)** · STATUS: OPEN — tag structure / cadence (catalog `cadences` field on the two rows).
- **What:** a badge URL resolving (or a transient-vs-real distinction) is a slow-moving property — it does not need per-PR verification the way a code/docs-alignment gate does. Paying ~127s of live-network latency on every single PR to re-confirm badges that change maybe once a milestone is poor ROI, and it also imports live-network flakiness into the pre-pr path (a badge provider hiccup can NOT-VERIFY/FAIL a PR that changed nothing badge-related).
- **Fix-sketch:** move both rows off `pre-pr` to `weekly` + `on-demand` (the pattern already used for other slow-moving live-network freshness rows) — weekly catches badge rot with alerting, on-demand covers a deliberate pre-release check, and per-PR runs drop ~127s + shed live-network flakiness. If a per-PR signal is still wanted, keep only the cheapest local-resolvable half (e.g. static badge-markup presence) on pre-pr and push the live-HTTP resolution to weekly. Re-measure pre-pr after the move to confirm the ~127s drop.
- **Effort:** trivial-small (~30min) — edit the `cadences` list on the two catalog rows via the proper mint path (never hand-edit the JSON) + re-run `--cadence pre-pr` to confirm the timing drop; verify weekly still grades them.
