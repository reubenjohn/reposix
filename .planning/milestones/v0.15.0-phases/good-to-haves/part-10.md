### GTH-V15-87 — zsh `${PIPESTATUS[0]}` is 1-indexed — a pipeline-exit-code grounding gotcha that bit a P124 leaf
- **Source:** NOTICED at P124 phase-close — a P124 leaf mis-read a pipeline exit code once via `${PIPESTATUS[0]}` in this project's zsh. · **Severity: LOW (grounding / footgun — no shipped defect, but a repeatable trap)** · STATUS: OPEN — tag docs / grounding.
- **What:** in bash, `${PIPESTATUS[0]}` is the FIRST pipeline element's exit (0-indexed); in this project's zsh the equivalent array (`pipestatus`) is **1-indexed**, so `${PIPESTATUS[0]}` mis-reads (empty / wrong element) and a guard like `[ "${PIPESTATUS[0]}" -ne 0 ]` silently mis-fires. A P124 leaf hit this once. Correct zsh forms: `${pipestatus[1]}` for the first element, or avoid the array entirely and verify via a non-piped `$?`.
- **Fix-sketch:** add a one-line shell-convention note to a grounding doc (e.g. `crates/CLAUDE.md` build-discipline section or a `scripts/` conventions note) — "zsh `pipestatus` is 1-indexed; use `${pipestatus[1]}` or a non-piped `$?`, never `${PIPESTATUS[0]}` for the first pipe stage in this repo's zsh" — so future leaves don't re-trip it. Fix-twice candidate (document the gotcha where the next agent reads it).
- **Effort:** trivial (<15min) — one doc line.

## From /gsd-quick CI-infra fix (pre-pr timeout band-aid, 260719-*, 2026-07-19)

### GTH-V15-93 — improve CI rust-cache hit rate so `quality gates (pre-pr)` cold builds don't approach the timeout wall (the REAL fix; the 15→28min raise landed the SAME commit is a band-aid)
- **Source:** FILED by a `/gsd-quick` leaf dispatched by L0 to stop the recurring `quality gates (pre-pr)` job cancellation on `main` — 4 `cancelled` ci.yml runs in ~2 days initially misdiagnosed as queue contention (see `.planning/notes` / STATE relief log `#65→#66`). Forensic re-read of job timing (not just run `conclusion`) found the real driver. · **Severity: MEDIUM-HIGH (CI health — currently masked by a band-aid, will recur as the workspace grows)** · STATUS: OPEN — tag code / CI infra.
- **What — the misdiagnosis:** GitHub reports a `timeout-minutes` kill as `conclusion: cancelled` with NO job-level `timed_out` flag, and `ci.yml`'s `concurrency: { cancel-in-progress: true }` block (`ci.yml:16-18`) ALSO produces `cancelled` runs when a push supersedes an in-flight one — so a timeout-wall hit and a benign supersede-cancel are indistinguishable from `conclusion` alone. Breakdown of the 4 cancelled runs: **1/4 benign** — run `29677548830` / commit `495b8357`, `cancel-in-progress` auto-cancel (superseded by the next push; every job in that run died together, consistent with a group cancel, not a per-job timeout). **3/4 real timeout-wall** — the `quality gates (pre-pr)` job ran ALONE to 15m51s–20m00s while every sibling job in the same run finished in minutes; it was killed at the then-15-min cap.
- **Warm-vs-cold proof:** commit `394ebc3a`'s pre-pr job — attempt 1: 20:00 (killed, cold rust-cache miss). attempt 2: 20:00 (killed, cold rust-cache miss again). attempt 3: **3m39s success** (warm cache hit). Other confirmed cold hits: run `29672608656` / commit `6472f020` = 15m51s; run `29671165231` / commit `28a8646b` = 20:00. A later green baseline (run `29677846998` / commit `592ae4c0`, 2026-07-19) measured pre-pr at a warm 4m06s — consistent with the ~3-4m warm-cache floor. The `cargo build --workspace --bins` "Warm cargo build" step (`ci.yml` pre-pr job, step `Warm cargo build`) is the step that pays the cold-compile cost; every other pre-pr step (mkdocs install, jq/sqlite3 install, `quality/runners/run.py --cadence pre-pr`) is fast and cache-insensitive.
- **The band-aid landed in this same commit:** `timeout-minutes: 15` → `28` on the `quality gates (pre-pr)` job (headroom over the observed ~20m cold-cache max) + explicit `timeout-minutes` added to the 3 sibling jobs that previously inherited GitHub's silent 360-min default (`test`: 6min, observed 2m28s-2m45s; `coverage`: 6min, observed 2m34s-2m53s; `shell-coverage`: 3min, observed 1m07s-1m18s — all ~2x observed, bounded fast-fail). This buys headroom and stops silent-default sibling jobs from running unbounded, but does NOT fix the underlying cache-miss rate — a sufficiently large cold miss (workspace growth over time) can still approach 28min, and a genuinely-cold miss still burns ~20min of CI wall-clock and $ before it even matters.
- **Fix-sketch (the REAL fix):** improve the `Swatinem/rust-cache@v2` hit rate for the `quality gates (pre-pr)` job specifically. Candidates worth investigating in order of likely leverage: (1) confirm the pre-pr job's cache key isn't being invalidated more often than necessary (e.g. by `Cargo.lock` churn from unrelated crates, or a `save-if:` misconfiguration causing every run to write a fresh cache instead of reusing); (2) consider sharing/warming the cache from the `clippy` or `test` job (which run earlier / in parallel and already prime a workspace build) via `Swatinem/rust-cache@v2`'s `shared-key` option, so pre-pr's `cargo build --workspace --bins` can hit a cache another job already populated in the SAME run instead of needing its own prior-run cache; (3) check whether `CARGO_INCREMENTAL: 0` (set globally at `ci.yml:13` for reproducible warnings-as-errors builds) is fighting the cache's ability to reuse artifacts across runs — verify empirically before touching this since it may be load-bearing for `-D warnings` determinism; (4) once the hit rate is measurably improved, consider whether `timeout-minutes: 28` can be safely lowered back down.
- **Effort:** medium (~2-3h) — needs actual cache-hit-rate instrumentation (e.g. temporarily log `Swatinem/rust-cache`'s cache-hit output across a handful of runs) before picking a fix, plus verification that any change doesn't regress `-D warnings` determinism or other jobs' cache behavior.
