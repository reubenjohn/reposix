## From P120 CLOSE Wave A (2026-07-17, credential-leak hardening)

### GTH-V15-62 — TWO credential redactors (`strip_url_userinfo` vs `redact_userinfo`) with a footgun split; unify into one canonical `reposix_core::http` redactor
- **Source:** NOTICED during P120 CLOSE Wave A while fixing WR-01/02/03 (three real credential leaks) · **Severity: MEDIUM (footgun / recurrence risk)** · STATUS: OPEN — tag security / api-ergonomics.
- **What:** credential redaction is split across two functions in two crates with a non-obvious, dangerous boundary. `reposix_core::http::strip_url_userinfo` parses ONE well-formed `http(s)` URL and — critically — **passes a `reposix::`-prefixed or otherwise-unparseable URL through UNCHANGED** (the outer scheme defeats `Url::parse`), so echoing a bus URL through it silently LEAKS the `?mirror=` credential. The correct tool for `reposix::`/bus/free-form/git-stderr strings is `reposix_remote::backend_dispatch::redact_userinfo` (a scanner that strips `://user:secret@` anywhere). The split is documented (`crates/CLAUDE.md` § Credential redaction) but the footgun is that the WRONG choice fails SILENTLY (compiles, "looks redacted", leaks). WR-02/WR-03 were originally scoped to use `strip_url_userinfo` and would have shipped a leak had the shape not been caught. The scanner also lives in `reposix-remote` (the git-helper crate), so every CLI/core caller must depend on `reposix-remote` just to redact — an inverted dependency smell.
- **Fix-sketch:** promote ONE canonical redactor into `reposix_core::http` (the natural home, next to the egress allowlist) that is scanner-based by default (strips userinfo from every `scheme://…@` authority in an arbitrary string, byte-exact on credential-free input), and either delete `strip_url_userinfo` or make it a thin alias that delegates. Migrate all call sites (`backend_dispatch.rs`, `bus_handler.rs`, `sync.rs`, `worktree_helpers.rs`) to the one function; drop `reposix-remote` from the redaction dependency path. Add a `#[deprecated]` shim window if needed. This removes the silent-wrong-choice footgun entirely.
- **Effort:** small-medium — one function move + call-site migration + re-point the `crates/CLAUDE.md` doctrine at the single canonical redactor.

### GTH-V15-66 — `resolve_cache_path`'s global `REPOSIX_CACHE_DIR` re-read races under threaded `cargo test` (no nextest process isolation on this host)
- **Source:** NOTICED during P120 CLOSE Wave B while auditing gate-coverage test infra · **Severity: LOW (pre-existing, not P120-introduced; test-infra fragility)** · STATUS: OPEN — tag test-infra / CI-parity.
- **What:** `reposix-cache::path::resolve_cache_path` reads the process-global `REPOSIX_CACHE_DIR` environment variable directly. `cargo-nextest` isolates each test into its own process (env mutations don't cross-contaminate), but this host does not have `cargo-nextest` installed, so the agent-ux gates and ad-hoc `cargo test` runs fall back to the default THREADED test harness, where all tests share ONE process and therefore one environment. Any two tests that both set/read `REPOSIX_CACHE_DIR` (directly or via a helper that shells out to `reposix init`/`attach` against a custom cache dir) can race: one test's `env::set_var` can be observed by another concurrently-running test before it restores the var, producing a flaky cache-path resolution. This is CI-parity fragility, not a proven-flaky test today — no failure was observed this session, but the hazard is structural and worsens as more cache-path-dependent tests are added.
- **Fix-sketch:** either (a) install `cargo-nextest` in CI and document it as the local-dev recommendation too (process isolation eliminates the race class entirely — the more mechanical fix), or (b) make `resolve_cache_path`'s env dependency injectable (accept an explicit `Option<&str>` override / a small `CacheDirSource` trait) so tests can pass an isolated value instead of mutating the shared process env — more invasive but removes the hazard even under threaded `cargo test`. (a) is lower-effort and is the precedent-matching choice (`quality/PROTOCOL.md`'s cadence docs already assume nextest for full-workspace runs).
- **Effort:** small (nextest install + CI wiring) to medium (env-injection refactor), depending on which fix track is chosen.
