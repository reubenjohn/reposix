### GTH-V15-51 — doc-alignment `parse_test` has no function-level pinning for non-Rust tests; `.py::fn` citations become a literal unresolvable path
- **Source:** NOTICED during P117 W2 Step C-i (implementing GTH-V15-49 Option B) · **Severity: MEDIUM** · STATUS: OPEN — tag docs-alignment / catalog-mint coarseness.
- **What:** `crates/reposix-quality/src/commands/doc_alignment.rs::parse_test` (`:1603-1621`) only splits `<file>::<fn>` into `TestRef::RustFn` when `file` ends in `.rs` (`is_rust` check at `:1605-1607`). For a non-Rust citation like `quality/gates/docs-repro/test_snippet_extract.py::test_pivot_counts_uncovered_not_raw_blocks`, the `::` split still happens (`s.rsplit_once("::")` at `:1604`) but `is_rust` is false, so the function falls through to `Ok(TestRef::File { file: PathBuf::from(s) })` at `:1618-1620` — `s` is the ENTIRE original string, including the literal `::test_pivot_...` suffix. That path does not exist on disk (`test.py::fn` is not a real file), so any `bind` call citing a Python/JS/shell test at function granularity is dead-on-arrival: `TestRef::File::compute_hash` (`crate::hash::file_hash`) will fail to find the file, or worse, silently hash a nonexistent-path sentinel depending on `file_hash`'s own error handling. The function's own doc comment (`:1590-1597`) documents this as intentional scope ("`::` form stays Rust-only") but does not warn callers that citing `<non-rust-file>::<fn>` is not merely "less precise" — it's a broken citation, not a graceful bare-file fallback.
- **Fix-sketch:** add pytest-node-style (or generic non-Rust) function-level pinning to `parse_test`: when `is_rust` is false but the string contains `::`, strip the `::<fn>` suffix and construct `TestRef::File { file: PathBuf::from(file) }` (bare file, dropping the fn) INSTEAD of `TestRef::File { file: PathBuf::from(s) }` (whole string) — this alone fixes the dead-path bug even without true function-body hashing. A fuller fix adds a new `TestRef` variant (e.g. `PyFn { file, fn_name }`) with a Python-specific body-hash strategy (`def <fn_name>(` line-scan, mirroring `test_body_hash`'s Rust AST walk but text-based) so non-Rust tests get the same drift-detection precision Rust tests already have. Minimal fix is the bare-file strip; full fix is the new variant + hasher.
- **Effort:** minimal fix small (~10-15 lines, strip-suffix-before-fallback); full fix medium (~1-2h, new `TestRef` variant + text-based Python fn-body hash + tests). Grader-instruction mitigation landed same-commit as this filing (see `.claude/skills/reposix-quality-doc-alignment/prompts/{grader,extractor}.md`): cite non-Rust tests as bare filenames, never `<file>::<fn>`, until this lands.
