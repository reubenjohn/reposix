### GTH-V15-05 — `resolve_import_parent()` silently degrades on ANY git error
- **Source:** part-02 (discovered-by P105 docs fix-twice lane, 2026-07-12 09:40) · **Severity: MEDIUM** · STATUS in intake: DEFERRED-TO-v0.15.0 (helper-hardening phase).
- **What:** `resolve_import_parent()` (`crates/reposix-remote/src/main.rs:400-419`) degrades to the parentless path (`None` → no `from`, no `deleteall`) on **any** git error, not just ref-absence. Two conflations: (1) the `rev_parse` closure returns `None` via `.ok()?` (`main.rs:407`) when the `git` spawn itself fails (binary missing / I-O error), swallowing a real environmental fault as "no parent"; (2) `!out.status.success()` (`main.rs:408`) treats every non-zero rev-parse exit as ref-absent. A future regression making rev-parse fail for a non-absence reason would silently re-open the RBF-LR-03 non-descendant "does not contain" abort with no operator-facing error. NOT addressed by the `bd5b9cb` disjoint-namespace fix (different failure mode).
- **Fix-sketch:** distinguish ref-absent (the legitimate parentless case: `rev-parse --verify --quiet <ref>` exits 1 with empty stdout AND the git spawn succeeded) from spawn / other rev-parse failures; on the latter, error the fetch loudly (`fatal:` + recovery hint) instead of degrading to a parentless overlay; keep the empty-stdout → `None` path for the genuine first-fetch case. Add a unit test injecting a non-absence git failure and asserting the fetch errors rather than emitting a parentless overlay. Small (<1h) but needs a cargo window. Belongs to the same `crates/reposix-remote` helper-hardening phase as GTH-V15-04 (residual verification) and GTH-V15-06.

### GTH-V15-57 — doc-alignment rows bind to LINE NUMBERS, so any edit above a row drifts it; this cost THREE full rebind cycles in P117 alone
- **Source:** NOTICED during the P117 W4 push-blocker fix (commit `0d4a0b7`'s 11-line `<video>` embed insertion into `docs/index.md:22` shifted 16 downstream row citations to `STALE_DOCS_DRIFT`, blocking pre-push) · **Severity: MEDIUM-HIGH** · STATUS: OPEN — tag docs-alignment / gate-design.
- **What:** `quality/catalogs/doc-alignment.json` rows cite `source.line_start`/`source.line_end` — an absolute line range within the target file — and the walker (`crates/reposix-quality/src/commands/doc_alignment.rs`) computes `source_hash` over exactly that byte range. Any commit that inserts or deletes lines ABOVE a bound row's citation shifts that row's true content downward or upward without touching the claim's actual text — but the walker has no way to distinguish "the claim moved" from "the claim changed," so it flags a pure line-shift as `STALE_DOCS_DRIFT` and blocks pre-push until a human/agent manually re-derives the new line range and re-binds. This happened **three separate times** in P117 alone, each a full push-block + manual rebind cycle:
  1. **W3, RBF-LR reword** — a banned-words fix (`ab0324b`) reworded `docs/index.md:167`'s prose, invalidating two rows citing that line (`blob-limit-teaching`, `push-conflict-detection`) even though only ONE clause's wording changed and the OTHER clause was untouched — the line-granularity binding couldn't distinguish which claim on the shared line actually drifted.
  2. **W3, banned-words reword (117-07)** — a Comments-footnote note-block conversion (`d57626a`) pushed `backend-capabilities-struct`'s citation from line 144 to 149 (+5), requiring a manual rebind purely for the shift.
  3. **W4, embed insertion (this lane, commit `0d4a0b7`)** — an 11-line `<video>` embed inserted after line 22 shifted ALL 16 downstream rows on `docs/index.md` by +11, requiring the full-file rebind executed in this same commit.
  Each cycle costs a manual line-number recompute (diffing old vs. new file, confirming byte-identical content, re-running `bind` per row) that is pure toil — the claim text and the test binding never actually changed, only the row's citation coordinates. This is the single highest-frequency doc-alignment toil source observed in P117 and will recur on every future edit that inserts/deletes lines above any bound row on a heavily-annotated page like `docs/index.md` (currently ~30+ bound rows).
- **Fix-sketch:** anchor row citations to something stable across line-shifting edits instead of an absolute line range:
  - **(a) Content-hash/fuzzy-match anchor (preferred):** store the claim's own text (or a normalized hash of it) as the primary anchor; on `walk`, locate the claim by searching for that content within the file rather than trusting the stored line range, and treat the row as drifted ONLY if the located text's hash differs from what was last bound — a pure line-shift then re-derives the new line range automatically and re-binds cleanly (or at minimum classifies as `LINE_SHIFT_ONLY`, distinct from `STALE_DOCS_DRIFT`, so the walker/wrapper could auto-rebind mechanically without a subagent judgment call).
  - **(b) Heading-relative anchor:** store `(nearest preceding heading slug, offset within section)` instead of an absolute file line; edits outside the cited section never drift the row, and edits within it still need a real rebind (correctly, since content actually changed).
  - **(c) Cheapest near-term mitigation (no schema change):** teach `walk` to diff the OLD bound text (recoverable from `source_hash`'s pre-image is not stored today, so this needs the claim's literal text stored in the row, which it partially already is via `claim`) against a fuzzy search in the CURRENT file; if an exact text match is found elsewhere in the file, auto-emit the line-shifted `bind` call instead of blocking pre-push. This is the smallest change that kills the recurring toil without redesigning the anchor scheme.
  Any of these directly targets the recurrence pattern: three rebind cycles in one phase, all doing the identical mechanical dance (confirm content byte-identical, recompute new line range, re-run `bind`) — the highest-leverage fix for the whole doc-alignment dimension's day-to-day friction.
- **Effort:** medium (a/b — new anchor scheme + migration of ~30+ existing rows + walker logic change) to small-medium (c — a fuzzy-match auto-rebind assist bolted onto the existing line-range scheme, no catalog schema change). Natural home: a docs-alignment framework-hardening lane, ideally before `docs/index.md`'s bound-row count grows further.
