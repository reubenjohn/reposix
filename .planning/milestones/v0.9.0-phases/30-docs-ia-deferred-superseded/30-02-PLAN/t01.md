```markdown
# Mental model in 60 seconds

reposix takes three ideas you already know and snaps them together. Read this page before `tutorial.md` if you want the conceptual bones first; read the tutorial first if you prefer to learn by doing.

## mount = git working tree

The folder reposix gives you is a git working tree. `ls` lists your tickets. `git status` shows what you changed. `git log` shows who edited what and when — the audit trail you already have, not a new one you have to build.

\`\`\`bash
cd ~/work/acme-jira
git status
# On branch main
# Changes not staged for commit:
#   modified:   issues/PROJ-42.md
\`\`\`

## frontmatter = schema

Every tracker has a schema — issue types, custom fields, labels, statuses. reposix puts all of it in the YAML frontmatter at the top of each file. Changing a field is changing one line of text.

\`\`\`yaml
---
id: PROJ-42
status: In Progress
assignee: alice@acme.com
labels: [backend, needs-review]
---
\`\`\`

## \`git push\` = sync verb

\`git push\` is the one verb you need to sync local edits with the remote tracker. The round-trip is not magic — it is a diff computed by git and dispatched through reposix as ordinary API calls. The next section ("How it works") shows the mechanism.

\`\`\`bash
git commit -am "close PROJ-42"
git push        # syncs frontmatter + body edits to the tracker
\`\`\`

---
```

```markdown
# reposix vs MCP and REST SDKs

!!! note "Stub — Wave 1 Copy plan will fill comparison table + numbers from `docs/why.md` token-economy benchmark."

reposix complements MCP servers and REST SDKs — it does not substitute for them. Use reposix for the 80% of tracker operations an agent does a hundred times a day: status changes, field edits, labels, comments, links. Keep your SDK or MCP server for the 20% that reposix cannot express: complex JQL, bulk imports, admin operations, reporting.

## When to use which

| Scenario | reposix | MCP | REST SDK |
|----------|:-------:|:---:|:--------:|
| Edit a ticket's status, assignee, or labels | ✓ | | |
| Bulk-import 10 000 tickets | | | ✓ |
| Run arbitrary JQL query | | ✓ | ✓ |
| Subscribe to webhooks | | ✓ | ✓ |
| Let an agent review + comment across 50 tickets | ✓ | | |

See [Why reposix](why.md#token-economy-benchmark) for the measured token-economy numbers that motivate this split.
```
