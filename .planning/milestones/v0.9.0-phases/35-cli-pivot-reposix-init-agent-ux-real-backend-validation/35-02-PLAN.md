# Phase 35 Plan 02 — Dark-factory regression test against simulator

## Goal

Prove the architecture's central thesis ("pure git, zero in-context learning") with an automatable test that does NOT require a Claude subagent. The test simulates an agent's stderr-reading behaviour with a small shell wrapper, exercising the three paths from the CONTEXT decisions: happy path, conflict-rebase cycle, blob-limit teaching.

## Tasks

### Task 35-02-A: `scripts/dark-factory-test.sh <backend>` shell wrapper

<read_first>
- `35-RESEARCH.md` §R2.
- Existing scripts in `/home/reuben/workspace/reposix/scripts/` (style: `set -euo pipefail`).
- `crates/reposix-cli/src/init.rs` (just added in 35-01).
</read_first>

<action>
1. Create `scripts/dark-factory-test.sh` that:
   - Takes `$1` as backend kind (`sim` is the only one supported in Phase 35; real backends pass through to be invoked from 35-03 tests).
   - For `sim`: spawns `reposix-sim --ephemeral --bind 127.0.0.1:7779` in the background, kills on EXIT trap.
   - Exports `REPOSIX_ALLOWED_ORIGINS=http://127.0.0.1:7779`.
   - Exports `REPOSIX_BLOB_LIMIT=200` (default).
   - Runs `reposix init sim::demo /tmp/dark-factory-$$/repo`.
   - Cd into the repo, runs `cat issues/<first>.md` (proof of clone), edits the issue body, `git add . && git commit -m 'add TODO' && git push`.
   - Greps for `git sparse-checkout` in stderr if any push fails — the literal "stderr-reading agent" simulation.
2. `chmod +x scripts/dark-factory-test.sh`.
3. The script must exit 0 on the happy path, surface the underlying failure on the unhappy path.
</action>

<acceptance_criteria>
- `bash scripts/dark-factory-test.sh sim` runs end-to-end with exit code 0 (when run as part of the phase verification).
- The script uses `set -euo pipefail`.
- An EXIT trap kills the sim child even on failure.
</acceptance_criteria>

<threat_model>
- The sim is bound to `127.0.0.1:7779`; the egress allowlist contains only that origin. No external network egress.
- The temp repo lives under `/tmp/dark-factory-$$` and is deleted by the EXIT trap.
</threat_model>

### Task 35-02-B: `crates/reposix-cli/tests/agent_flow.rs` integration test

<read_first>
- `35-RESEARCH.md` §R2.
- `crates/reposix-cli/tests/cli.rs` (assert_cmd patterns + workspace_root resolution).
</read_first>

<action>
1. Create `crates/reposix-cli/tests/agent_flow.rs` with three `#[test]` fns, all `#[ignore]`-gated (require git on PATH):
   - `dark_factory_sim_happy_path` — clone via `reposix init sim::<slug>`, inspect git config, edit and push.
   - `dark_factory_sim_conflict_path` — happy clone, mutate the same issue via direct REST POST against the sim, push fails with `error refs/heads/main fetch first` (the canned message from Phase 34), agent runs `git pull --rebase`, retries push, succeeds.
   - `dark_factory_sim_blob_limit_path` — `REPOSIX_BLOB_LIMIT=3` env var, attempt a checkout that requests >3 blobs, capture stderr, assert it contains `git sparse-checkout` literally (the Phase 34 dark-factory teaching string), run `git sparse-checkout set issues/<one>.md`, retry, succeeds.
2. Use `tempfile::TempDir` for repos, `reposix_cli::sim::SimProcess::spawn_ephemeral` for sim.
3. Pin a free port via `tokio::net::TcpListener` bind-and-drop *or* hardcode `127.0.0.1:7779` (test-isolated via `--test-threads=1`); the existing pattern uses hardcoded ports so we follow suit.
4. Each test runs `cargo build -p reposix-remote && cargo build -p reposix-cli` implicitly via `assert_cmd::Command::cargo_bin`. Add `git-remote-reposix` binary path to PATH for the test invocation by setting `env!("CARGO_BIN_EXE_git-remote-reposix")`'s parent.
</action>

<acceptance_criteria>
- `cargo test -p reposix-cli --test agent_flow -- --ignored` passes all three tests when git is on PATH.
- The conflict test asserts the literal `fetch first` error string (regression for Phase 34's canned status).
- The blob-limit test asserts the literal `git sparse-checkout` substring in stderr (regression for Phase 34's teaching mechanism).
</acceptance_criteria>

<threat_model>
- Tests bind sim to localhost only.
- Test fixtures (issue bodies) are tainted-by-default content; we never route them into shell expansion (only into `git add` operating on file paths under our control).
- The temp dir is dropped by `tempfile::TempDir`.
</threat_model>
