## Subprocess coverage: investigated and confirmed working

A prior version of this doc claimed `cargo llvm-cov` did not aggregate subprocess profile data — that claim was **wrong**. The investigation on 2026-04-27 (branch `cov-subprocess`) verified:

- `cargo-llvm-cov` sets `LLVM_PROFILE_FILE` on the test process. `assert_cmd::Command::cargo_bin(...)` propagates the env var to spawned children. Each subprocess writes its own `cov-subprocess-<pid>-<bin-hash>_0.profraw`.
- All workspace bins (incl. `target/llvm-cov-target/debug/git-remote-reposix`) are added to the `-object` list passed to `llvm-cov export`. The merge step picks up every profraw under `target/llvm-cov-target/`.
- Direct inspection (`llvm-profdata show --topn=N` on a `5545402776965270270`-hash profraw — the helper-bin signature) confirmed instantiation symbols for `git_remote_reposix::handle_export`, `real_main`, `proxy_one_rpc`, `send_advertisement` were all present, with non-zero counts when the test exercised them.

The previously-low file numbers on `stateless_connect.rs` (42.2%) and `main.rs` (79.6%) were not caused by lost subprocess data — they were caused by **no test exercising the `command=stateless-connect` tunnel**. `bulk_delete_cap.rs` and `push_conflict.rs` only drive the `export` (push) path; `protocol.rs` only drives the `capabilities` advertisement turn. The `partial_clone_against_sim_is_lazy` placeholder in `tests/stateless_connect.rs:85-104` is a `panic!()` skeleton awaiting Phase 32 CI infra.

The new `stateless_connect_e2e.rs` (3 scenarios) closes the gap by driving the protocol-v2 turn directly via stdin pkt-lines — no `git` binary required, no special CI image, no `#[ignore]` gating. Result: stateless_connect.rs 42.2% → 75.3%, main.rs 79.6% → 82.9%.

Methodology lesson (per CLAUDE.md "fix it twice"): before claiming subprocess data is lost, verify with `llvm-profdata show <profraw>` that the helper-bin profraw exists AND contains zero counts on the function in question. Zero counts means the helper bin was invoked but the test never reached the function — write a test that does — not "subprocess coverage is broken."

## Lines explicitly classified IRRELEVANT (do not test)

Per the audit scope, the following uncovered lines are excluded:

- `cli` argument parsing in `crates/reposix-cli/src/main.rs` and helpers.
- `Display` / `Debug` / `From` impls in `crates/reposix-core/src/error.rs`.
- Real-backend connectors (`reposix-github`, `reposix-confluence`, `reposix-jira`) gated on credential env vars — coverage there is exercised by `--ignored` tests with real creds, which is the explicit threat-model constraint (CLAUDE.md OP-1: "fail-closed by default").
- Defensive `tracing::warn!` branches in `reposix-cache::audit` (every helper has `if let Err(e) = res { warn!(...) }` for SQL failure). These are best-effort logging branches per the file-level docstring (§Atomicity, line 4: "audit-row write failures log WARN via `tracing::warn` but do NOT poison the caller's flow"). Forcing SQL failure to cover them would require an in-process fault injection that is not worth the test surface.
- Unreachable error formatting closures in `reposix-core::audit` (`load_schema`, `enable_defensive`, `open_audit_db`) where the failure path requires a corrupted SQLite build that does not occur in any reachable configuration.
