## NICE-TO-HAVE gaps (in priority order)

These would tighten the file-level numbers and remove ambiguity around subprocess coverage. None of them gate ship.

1. **Direct unit tests for 5 cache-audit helpers** — `crates/reposix-cache/src/audit.rs:86-112,120-177,313-334`.
   Functions `log_helper_connect`, `log_helper_advertise`, `log_helper_fetch`, `log_helper_fetch_error`, `log_helper_backend_instantiated` have zero direct unit-test coverage. They are exercised by the helper-binary subprocess in `agent_flow.rs` (`#[ignore]`-gated) but not by default `cargo test`. Adding 5 trivial "insert-then-query" tests in the existing `mod tests` block would lift cache audit from 78.9% → ~95%. Pattern already established by `log_blob_limit_exceeded_inserts_row` at `audit.rs:563-577`.

2. **Aggregate subprocess coverage for helper-binary tests** — methodology fix.
   `cargo llvm-cov --workspace --no-clean -- --include-ignored` plus `LLVM_PROFILE_FILE` propagation through `assert_cmd` would let `bulk_delete_cap.rs` and `push_conflict.rs` contribute to the report. Without this, the 42% on `stateless_connect.rs` and the 79.6% on `reposix-remote/src/main.rs` will continue to misrepresent actual coverage of boundaries 4 and 5.

3. **Direct test for `HttpClient::post` / `patch` / `delete` allowlist gate** — `crates/reposix-core/src/http.rs:307-325`.
   `request()` and `get()` are heavily tested; the convenience wrappers for the other three verbs are not. They share the same `request_with_headers_and_body` body, so the gate cannot drift, but a one-liner per verb would make the contract explicit. Lifts http.rs from 92% → ~96%.

## Lines explicitly classified IRRELEVANT (do not test)

Per the audit scope, the following uncovered lines are excluded:

- `cli` argument parsing in `crates/reposix-cli/src/main.rs` and helpers.
- `Display` / `Debug` / `From` impls in `crates/reposix-core/src/error.rs`.
- Real-backend connectors (`reposix-github`, `reposix-confluence`, `reposix-jira`) gated on credential env vars — coverage there is exercised by `--ignored` tests with real creds, which is the explicit threat-model constraint (CLAUDE.md OP-1: "fail-closed by default").
- Defensive `tracing::warn!` branches in `reposix-cache::audit` (every helper has `if let Err(e) = res { warn!(...) }` for SQL failure). These are best-effort logging branches per the file-level docstring (§Atomicity, line 4: "audit-row write failures log WARN via `tracing::warn` but do NOT poison the caller's flow"). Forcing SQL failure to cover them would require an in-process fault injection that is not worth the test surface.
- Unreachable error formatting closures in `reposix-core::audit` (`load_schema`, `enable_defensive`, `open_audit_db`) where the failure path requires a corrupted SQLite build that does not occur in any reachable configuration.
