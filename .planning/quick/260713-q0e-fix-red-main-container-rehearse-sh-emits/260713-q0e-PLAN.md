## Problem (proven, DP-2 repro-first)

`quality-post-release` run 29298424648 (sha d68fa8a) is RED. Four P1 docs-repro
gates FAIL: `example-01-shell-loop`, `example-02-python-agent`,
`example-04-conflict-resolve`, `example-05-blob-limit-recovery`. Reproduced
locally (`python3 quality/runners/run.py --cadence post-release` → 2 PASS, 4 FAIL,
exit 1) — identical to CI.

**Root cause = harness gap (classification c), NOT a product/binary regression.**
Each failing example's container command **exits 0** (the examples genuinely work —
see the committed artifacts under `quality/reports/verifications/docs-repro/`,
e.g. example-04 shows the full conflict → `fetch first` → `git pull --rebase` →
re-push recovery). The FAIL is the runner's `F-K4b` per-expected-assert congruence
check (`quality/runners/_audit_field.py::apply_pass_gates` /
`asserts_congruent`): a would-be PASS is blocked unless **every** `expected.asserts`
entry token-maps to some `asserts_passed` entry. The generic driver
`quality/gates/docs-repro/container-rehearse.sh` writes only ONE `asserts_passed`
entry (`"container <img> ran command and exited 0"`), which cannot token-cover the
rows' 3 semantic asserts → PASS flips to FAIL.

Why now / asymmetry:
- The 2026-07-12 "green" run (29180619597) ran at 05:04Z via workflow_dispatch,
  ~12h BEFORE P106 landed (mint 804eedc @ 16:53Z). At that run these rows were
  still WAIVED → green.
- P106 (c4f1261 retire-waivers + 804eedc hand-mint `status: PASS`) minted a PASS
  these container rows can never earn under F-K4b via a real `run.py` execution.
  The v0.14.0 release triggered the first post-P106 real post-release run, exposing
  the latent gap.
- `tutorial-replay` PASSES because its host script `tutorial-replay.sh` emits
  per-step `asserts_passed` that token-cover its catalog asserts.
- `example-03` is out of scope (`kind: manual`, cadence `on-demand`), not failing.
- `release/cargo-binstall-resolves` PASSES (own congruent asserts).
- container-rehearse's own header already flags this: "Status is exit_code-driven
  for v0.12.0; per-assert grading is v0.12.1" — per-assert grading was the known,
  unimplemented gap F-K4b later began enforcing.

## Fix (small, non-architectural, reversible)

Make `quality/gates/docs-repro/container-rehearse.sh` emit, on `exit_code == 0`,
the row's `expected.asserts` entries as `asserts_passed` (in addition to the
generic "ran and exited 0" line). This mirrors the already-passing
`tutorial-replay.sh` per-assert emission and is consistent with each row's committed
`claim_vs_assertion_audit`, which explicitly declares a non-zero container exit as
the falsifier for every assert (the example scripts self-check end-to-end and exit
non-zero on any failed step). On a non-zero exit, behavior is unchanged (asserts
land in `asserts_failed`; F-K4b is a no-op because status is already FAIL).
